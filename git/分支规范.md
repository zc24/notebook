# Git 分支命名规范

整理业界主流的 Git 分支命名规范，参考了 GitHub、GitLab、Google、Atlassian、Microsoft 以及主流开源项目（Linux Kernel、Kubernetes、React、Vue 等）的实践。

---

## 一、核心原则（通用）

| 原则 | 说明 |
|---|---|
| **小写** | 全部使用小写字母，避免大小写混用带来的跨平台问题（Windows/macOS 不区分大小写） |
| **分隔符用 `-` 或 `/`** | `-` 用于词与词之间，`/` 用于命名空间分层（如 `feature/login`） |
| **禁用空格、下划线、特殊字符** | 避免 `_`、空格、`@`、`#` 等 |
| **简短且有语义** | 一般控制在 **50 字符**以内，能见名知意 |
| **不要以 `/` 或 `-` 结尾** | 否则部分 Git 工具会报错 |
| **不要使用纯数字** | 如 `1234`，应改为 `issue-1234` |
| **关联 Issue / Ticket ID** | 便于追溯，如 `feature/JIRA-123-add-login` |

## 二、主流命名模式

### 1. Git Flow（Vincent Driessen 提出，最经典）

```text
main                  # 生产分支
develop               # 开发主分支
feature/<name>        # 新功能
release/<version>     # 预发布
hotfix/<name>         # 紧急修复
bugfix/<name>         # 普通 bug 修复
support/<version>     # 长期支持
```

示例：

```text
feature/user-authentication
release/1.2.0
hotfix/login-crash
```

### 2. GitHub Flow（GitHub 官方推荐，最轻量）

只有 `main` + 短生命周期的功能分支：

```text
<type>/<short-description>
```

示例：

```text
add-search-api
fix-login-bug
update-readme
```

### 3. GitLab Flow（带环境分支）

```text
main
pre-production
production
feature/<name>
```

### 4. Conventional Branch（社区规范，类比 Conventional Commits）

`<type>/<description>` 格式，类型受限：

| 前缀 | 用途 |
|---|---|
| `feature/` | 新功能 |
| `bugfix/` | bug 修复 |
| `hotfix/` | 紧急生产修复 |
| `release/` | 发布分支 |
| `refactor/` | 重构 |
| `docs/` | 文档 |
| `test/` | 测试 |
| `chore/` | 杂项、构建、依赖升级 |
| `experiment/` | 实验性 |

示例：

```text
feature/oss-upload-retry
bugfix/report-pagination-off-by-one
chore/upgrade-goframe-v2.7
```

## 三、大厂 / 开源项目实践

### Google（Chromium / Android）

- 主分支：`main`（已从 `master` 迁移）
- 个人分支：`<username>/<feature>`，如 `alice/refactor-render`
- 集成分支：`integration/<topic>`

### Microsoft（.NET / VSCode）

- `main`、`release/<version>`、`dev/<alias>/<topic>`
- 例：`dev/johndoe/fix-tree-view`

### Atlassian（推荐与 Jira 集成）

```text
<type>/<JIRA-KEY>-<description>
```

示例：

```text
feature/PROJ-123-add-payment
bugfix/PROJ-456-fix-null-pointer
```

### Linux Kernel

- 维护者分支：`<maintainer>/<topic>`，如 `torvalds/linux`
- 主题分支：`for-next`、`for-linus`、`fixes`

### Kubernetes

- `master` / `main`
- `release-<minor>`，如 `release-1.29`
- `dev-<feature>`

### React / Vue / Vite

- `main`
- `<version>-stable`，如 `18-stable`、`v2`
- `feat/<topic>`、`fix/<topic>`

### Angular

- `main`
- `<major>.<minor>.x`（如 `17.0.x`）作为补丁分支
- PR 分支：`<type>/<description>`

## 四、推荐模板（综合最佳实践）

```text
<type>/<ticket-id>-<kebab-case-description>
```

完整示例：

```text
feature/PROJ-1024-add-user-login
bugfix/PROJ-2048-fix-timezone-offset
hotfix/PROJ-3001-mq-consumer-leak
refactor/PROJ-4002-extract-payment-logic
chore/PROJ-5003-bump-dependencies
release/v1.5.0
docs/update-api-spec
```

## 五、需要避免的反例

| 反例 | 问题 |
|---|---|
| `Feature_Login` | 大写 + 下划线 |
| `fix bug` | 含空格 |
| `张三的分支` | 中文 / 个人色彩 |
| `temp`、`test`、`new` | 无信息量 |
| `feature/` | 空描述 |
| `feature/this-is-a-very-long-branch-name-that-describes-everything-in-detail` | 过长 |
| `master`（在新仓库中） | 业界已普遍迁移到 `main` |

---

## 六、工程化落地建议

1. **在仓库根目录 `CONTRIBUTING.md` 中明确写出规范**。
2. **CI 中加分支名校验**（GitHub Actions / GitLab CI 用正则）：

   ```yaml
   pattern: '^(feature|bugfix|hotfix|release|refactor|docs|test|chore)\/[A-Z]+-\d+-[a-z0-9-]+$'
   ```

3. **配合 Conventional Commits + Husky/pre-commit** 做提交信息校验。
4. **保护主干分支**：`main`、`develop`、`release/*` 开启 Protected Branch + Required Review。
5. **过期分支自动清理**：合并后自动删除 source branch（GitHub/GitLab 都支持）。
