## 一、核心原则

- **小写 + kebab-case**：全部使用小写字母，单词之间用 `-` 连接。Windows / macOS 文件系统默认不区分大小写，混用大小写会导致跨平台同步异常。
- **`/` 用于命名空间分层**：如 `feature/login`、`release/1.2.0`，便于在 GUI 客户端中按目录折叠展示。
- **禁用空格、下划线、中文、特殊字符**：避免 `_`、空格、`@`、`#`、`中文` 等，部分 CI 工具和 shell 脚本对这些字符处理不友好。
- **简短且有语义**：建议总长度 **≤ 50 字符**，能见名知意。
- **不要以 `/` 或 `-` 结尾**：部分 Git 工具会报错或产生隐藏空命名空间。
- **不要使用纯数字**：如 `1234` 应改为 `issue-1234`，避免和 commit hash、tag 混淆。
- **关联 Issue / Ticket ID**：在分支名中带上工单号（如 `feature/PROJ-123-add-login`），便于追溯需求来源、自动关联 PR 与工单。
- **短生命周期 + 合并即删**：功能分支应在数天～两周内合入主干并删除，避免长期存在的"僵尸分支"带来合并地狱。

## 二、工作流模型选型

不同的分支模型适用于不同的发布节奏与团队规模，**先选模型，再谈命名**。

### 2.1 Git Flow（重型，多版本并行）

由 Vincent Driessen 在 2010 年提出，最经典的双主干模型，适合需要**同时维护多个发布版本**的场景（如桌面软件、SDK、有线下交付的 B 端产品）。

```text
main                  # 生产分支，每次合并即一次发布，打 tag
develop               # 开发主分支，集成所有已完成的 feature
feature/<name>        # 新功能，从 develop 切，合回 develop
release/<version>     # 预发布稳定化，从 develop 切，合回 main + develop
hotfix/<name>         # 紧急生产修复，从 main 切，合回 main + develop
bugfix/<name>         # 普通 bug 修复（非紧急），通常等同 feature 分支
support/<version>     # 长期支持分支（LTS）
```

示例：

```text
feature/user-authentication
release/1.2.0
hotfix/login-crash
support/1.x
```

> **取舍**：分支层级清晰，但流程繁琐、合并冲突高，已被多数互联网团队抛弃。Vincent Driessen 本人也在 [2020 年更新](https://nvie.com/posts/a-successful-git-branching-model/) 中明确表示："如果你在做持续交付的 Web 应用，请考虑 GitHub Flow"。

### 2.2 GitHub Flow（轻量，持续部署）

GitHub 官方推荐，只有一条长期分支 `main` + 短生命周期的特性分支，适合**持续部署的 Web 应用 / SaaS**。

**原教旨：扁平命名，无 `type/` 前缀**。GitHub Flow [官方文档](https://docs.github.com/en/get-started/using-github/github-flow) 并不要求分支前缀，只要求"用一个描述性的名字"——这与 Git Flow 的 `feature/...`、`release/...`、`hotfix/...` 三类语义分支形成鲜明对比：

```text
main                  # 唯一长期分支，永远可发布
<short-description>   # 扁平命名，描述要做什么即可，合并即删
```

原教旨示例：

```text
add-search-api
fix-login-bug
update-readme
refactor-payment-flow
increase-test-coverage
```

**现代实践：借用 Conventional Branch 的 `<type>/<description>`**。多数团队仍会引入 type 前缀（参见 [3.1 通用模板](#31-通用模板)），让分支名携带工作性质，便于按 type 过滤、自动化、和 commit type 对齐：

```text
feat/add-search-api
fix/login-bug
docs/update-readme
```

> **取舍**：流程极简，依赖强大的 CI / CD 与自动化测试做兜底；不适合需要多版本并行维护的场景。
> 命名风格上，原教旨派 vs 现代实践派都常见，本规范 [第三节](#三分支命名格式) 推荐后者。

### 2.3 GitLab Flow（多环境）

在 GitHub Flow 基础上增加**环境分支**，适合有多套部署环境（staging / pre / prod）的团队。

```text
main                  # 开发主干
pre-production        # 预生产环境分支
production            # 生产环境分支
feature/<name>        # 功能分支
```

发布流程是单向上游推送：`feature/* → main → pre-production → production`，每次部署即一次"上游 merge 到下游环境分支"。

### 2.4 Trunk Based Development（主干开发）

所有开发者直接向 `main` 提交（小步快跑），或使用极短的 **short-lived branch**（< 1 天）。Google、Facebook、Netflix 等大厂内部主流模式，依赖 **Feature Flag** 控制功能可见性。

```text
main                  # 唯一主干，所有改动几乎实时合入
<user>/<topic>        # 个人短分支（可选，存在 < 1 天）
release/<version>     # 仅在需要发布时从 main 拉取，只用于打补丁
```

**关于多环境**：TBD **仍需 `dev` / `staging` / `prod` 等多套物理环境**，只是**不用"分支"来承载环境**（这是 TBD 与 GitLab Flow 最核心的区别）：

| 维度 | GitLab Flow | Trunk Based |
|------|-------------|-------------|
| 环境如何承载 | 用**分支**承载（`main` / `pre-production` / `production`），代码通过 merge 在分支间流转 | 用**部署流水线**承载，同一构建产物按 `dev → staging → prod` 顺序晋级 |
| 各环境代码差异 | 由分支差异决定（可能不一致） | 始终是同一个 commit / 同一份构建产物（Promotion Model） |
| 功能开关 | 一般依赖分支隔离 | 强制依赖 **Feature Flag**（LaunchDarkly / Unleash / 自建） |
| 环境配置 | 可写在代码里按分支区分 | 必须外置（ConfigMap / env vars / Vault） |
| 灰度 / 金丝雀 | 通常无 | 标配（按用户百分比、按 region 滚动放量） |

> 一句话：**GitLab Flow 用"切分支"换环境，Trunk Based 用"推流水线"换环境**。前者门槛低、上手快；后者要求强 CI/CD 与 Feature Flag 体系，但能做到分钟级发布与灰度。

### 2.5 选型建议

| 团队 / 场景 | 推荐模型 | 理由 |
|------------|---------|------|
| Web SaaS、持续部署 | GitHub Flow / Trunk Based | 流程轻、迭代快，配合 Feature Flag |
| 多环境部署、CI/CD 体系较弱 | GitLab Flow | 环境分支天然映射部署阶段，门槛低 |
| 多环境部署、CI/CD 体系成熟 | Trunk Based + 部署流水线 + Feature Flag | 单一构建产物逐级晋级，环境间零代码差异（参见 [2.4 关于多环境](#24-trunk-based-development主干开发)） |
| SDK / 桌面软件 / 需多版本并行维护 | Git Flow | release / support 分支支持长期版本 |
| 超大规模团队（百人以上单仓） | Trunk Based + Feature Flag | 避免长期分支冲突，依赖强 CI |
| 小团队 / 创业项目 | GitHub Flow | 上手最快，规则最少 |

## 三、分支命名格式

> **⚠️ 先厘清两层语义**：分支命名里出现的"前缀"（带 `/`）实际上有两种来源，二者**容易混淆**：
>
> | 前缀类别 | 来源 | 语义 | 典型例子 |
> |---------|------|------|---------|
> | **工作流分支** | Git Flow / GitLab Flow 等工作流模型 | 描述分支在**工作流中的角色**（来源 / 去向 / 合并策略不同），由模型决定，长期或中期存在 | `main`、`develop`、`release/v1.5.0`、`support/1.x`、`production` |
> | **命名 type 前缀** | Conventional Branch（借自 Conventional Commits） | 描述本次工作的**性质**（功能 / 修复 / 重构 / 文档…），分支本身仍是普通**短期分支**，合入即删 | `feature/...`、`fix/...`、`perf/...`、`docs/...` |
>
> `feature/`、`bugfix/`、`hotfix/`、`release/` 这几个名字在两类语义下都会出现，需要根据上下文判断：
>
> - 在 **Git Flow** 中，`release/v1.5.0` 是**中长期工作流分支**，承担"发布稳定化"角色（详见 [第四节](#四长期分支与保护策略)），从 `develop` 切出、合回 `main` + `develop`；
> - 在 **GitHub Flow / Trunk Based** 中，`release/v1.5.0` 只是借用了 `release` 这个 type 前缀的**短期分支**，仅命名表达"这次是发布相关工作"，合入主干即删。
>
> 本节（三）描述的是 **命名 type 前缀**（适用于所有短期分支）； [第四节](#四长期分支与保护策略) 描述的是 **工作流分支**（长期 / 中期分支与保护策略）。两者**配合使用**：先按工作流模型确定长期分支，再用命名规范命名所有短期分支。

### 3.1 通用模板

推荐使用 **Conventional Branch** 风格（类比 Conventional Commits），统一格式为：

```text
<type>/<ticket-id>-<kebab-case-description>
```

字段说明：

| 字段 | 必填 | 说明 |
|------|------|------|
| `type` | 是 | 分支类别，见下方[速查表](#32-type-速查表)，全小写 |
| `ticket-id` | 否（强烈推荐） | 关联的工单号，如 `PROJ-123`、`#1234`，便于追溯 |
| `description` | 是 | 一句话描述本次工作内容，**kebab-case**，**≤ 30 字符** |

最小示例：

```text
feature/add-user-login
fix/login-crash
```

带工单号示例：

```text
feature/PROJ-1024-add-user-login
bugfix/PROJ-2048-fix-timezone-offset
hotfix/PROJ-3001-mq-consumer-leak
refactor/PROJ-4002-extract-payment-logic
chore/PROJ-5003-bump-dependencies
release/v1.5.0
docs/update-api-spec
```

### 3.2 type 速查表

分支 `type` 与 [`commit规范.md`](./commit规范.md) 中的 commit `type` **一一对齐**，避免一套人员记两套词汇。

| type | 中文含义 | 使用场景 | 生命周期 |
|------|---------|---------|---------|
| `feature` / `feat` | 新功能 | 新增对用户可见的功能 | 短（数天～两周） |
| `bugfix` / `fix` | Bug 修复 | 修复非紧急的 bug | 短 |
| `hotfix` | 紧急修复 | 修复线上 P0 / P1 故障，走特殊发布流程 | 极短（数小时～1 天） |
| `release` | 发布分支 | 版本稳定化、打 tag 前的最后一道闸 | 短 |
| `refactor` | 重构 | 既不是新功能也不是修 bug 的代码改动 | 短 |
| `perf` | 性能优化 | 算法 / SQL / 缓存 / 内存等提升性能的改动 | 短 |
| `docs` | 文档 | 只改文档 / 注释 | 短 |
| `test` | 测试 | 新增 / 修改测试代码（单测、集成、e2e） | 短 |
| `build` | 构建 | 影响构建系统或外部依赖（`go.mod`、`package.json`、`Dockerfile`、`Makefile` 等） | 短 |
| `ci` | CI 配置 | CI 配置文件和脚本的改动（GitHub Actions、GitLab CI 等） | 短 |
| `chore` | 杂项 | 不属于以上任何一种的日常维护（编辑器配置、`.gitignore` 等） | 短 |
| `experiment` / `spike` | 实验性 | 技术验证 / 原型，**不保证合入主干** | 短 |
| `support` | 长期支持 | 维护旧版本的 LTS 分支（仅 Git Flow） | 长（数月～数年） |

> type 集合应**保持精简**，团队内统一一份枚举；建议在 CI 中通过正则校验，避免随意发挥。如果团队倾向 Angular 最新版的精简集，可省略 `chore`，把杂项细分到 `build` / `ci` / `refactor` / `docs` / `test` 之一。

### 3.3 description 写法

- **kebab-case**：单词全小写，用 `-` 连接，如 `add-user-login`。
- **祈使句 / 动词开头**：`add-xxx` / `fix-xxx` / `remove-xxx`，而不是 `added-xxx` / `xxx-fixed`。
- **写"做什么"，不写"在哪做"**：分支名是"动作"而非"路径"，`feature/login` 优于 `feature/src-pages-login`。
- **长度 ≤ 30 字符**：超长描述应拆细任务或简化措辞，细节写到 PR description 中。

对比：

| 不规范 | 规范 |
|--------|------|
| `Feature_Login` | `feature/add-login` |
| `feature/张三的登录` | `feature/PROJ-123-add-login` |
| `feature/this-is-a-very-long-branch-name-that-describes-everything-in-detail` | `feature/PROJ-123-add-login` |
| `feature/login_v2` | `feature/refactor-login` |
| `bugfix/fixed-bug` | `bugfix/PROJ-456-fix-token-refresh` |

## 四、长期分支与保护策略

| 分支 | 类型 | 保护策略 |
|------|------|---------|
| `main`（原 `master`） | 长期，唯一可发布 | **必须**开启 Protected Branch：禁止 push、要求 PR + Code Review + CI 通过 |
| `develop` | 长期，集成开发（仅 Git Flow） | 同上 |
| `release/*` | 中期，发布稳定化 | 同上，且只接受 bugfix 类合并 |
| `support/*` | 长期，LTS 维护 | 同上 |
| `feature/*` / `bugfix/*` / `chore/*` 等 | 短期，合入即删 | 无需保护，但 PR 合并后**自动删除** source branch |

## 五、常见场景示例

新功能开发：

```text
feature/PROJ-1024-add-user-login
feat/oss-upload-retry
```

Bug 修复（按紧急程度分流）：

```text
bugfix/PROJ-2048-fix-timezone-offset      # 普通 bug
hotfix/PROJ-3001-mq-consumer-leak          # 线上紧急
```

重构 / 性能：

```text
refactor/PROJ-4002-extract-payment-logic
perf/cache-normalized-query
```

构建 / CI / 依赖升级：

```text
build/upgrade-goframe-v2.7
build/migrate-to-vite-5
ci/add-codecov-upload
chore/update-gitignore
```

文档 / 测试：

```text
docs/update-api-spec
test/add-coupon-edge-cases
```

发布：

```text
release/v1.5.0
release/2025-q4
```

实验性技术验证：

```text
experiment/try-rust-rewrite
spike/llm-rag-poc
```

## 六、大厂 / 开源项目对照

| 项目 / 公司 | 主分支 | 命名风格 | 典型示例 |
|------------|--------|---------|---------|
| **Google（Chromium / Android）** | `main` | 个人 / 主题分支 | `alice/refactor-render`、`integration/<topic>` |
| **Microsoft（.NET / VSCode）** | `main` | `dev/<alias>/<topic>` | `dev/johndoe/fix-tree-view`、`release/<version>` |
| **Atlassian（Jira 集成）** | `main` | `<type>/<JIRA-KEY>-<desc>` | `feature/PROJ-123-add-payment` |
| **Linux Kernel** | `master` | 维护者 / 主题分支 | `torvalds/linux`、`for-next`、`for-linus`、`fixes` |
| **Kubernetes** | `master` / `main` | 版本 / 特性分支 | `release-1.29`、`dev-<feature>` |
| **React / Vue / Vite** | `main` | 版本稳定 + type 前缀 | `18-stable`、`v2`、`feat/<topic>`、`fix/<topic>` |
| **Angular** | `main` | 版本补丁 + PR 分支 | `17.0.x`、`<type>/<description>` |
| **GitHub Dependabot** | - | 自动生成 | `dependabot/npm_and_yarn/lodash-4.17.21` |

> **结论**：大厂普遍采用 **`<type>/<description>`** 或 **`<type>/<ticket>-<description>`** 风格，差异主要在是否带工单号、是否带个人前缀。新项目直接采用 [3.1](#31-通用模板) 推荐模板即可。

## 七、常见反例

| 反例 | 问题 | 正确写法 |
|------|------|---------|
| `Feature_Login` | 大写 + 下划线，跨平台风险 | `feature/add-login` |
| `fix bug` | 含空格，部分工具会报错 | `bugfix/PROJ-123-fix-token-refresh` |
| `张三的分支` | 中文 + 个人色彩，无追溯性 | `feature/PROJ-123-add-login` |
| `temp` / `test` / `new` / `tmp` | 无信息量，团队无法判断用途 | `experiment/llm-rag-poc` |
| `feature/` | 空描述 | `feature/PROJ-123-add-login` |
| `feature/this-is-a-very-long-branch-name-that-describes-everything-in-detail` | 过长，难以阅读和补全 | `feature/PROJ-123-add-login` |
| `1234` | 纯数字，易与 hash / tag 混淆 | `bugfix/issue-1234-fix-login` |
| `master`（新仓库） | 业界已普遍迁移到 `main` | `main` |
| `feature/login_v2` | 用 `v2` 表达"重做"语义不清 | `refactor/login` 或 `feature/redesign-login` |
| `merge-main-into-feature` | 合并操作不应作为分支名 | rebase / merge 操作不需要新分支 |

## 八、工程化落地建议

1. **在 `CONTRIBUTING.md` 中明确规范**：把本规范精简版写入仓库根目录的 `CONTRIBUTING.md`，新成员入职第一份必读文档。
2. **CI 中强制校验分支名**：通过正则在 PR 打开时卡掉不规范分支。GitHub Actions 示例：

   ```yaml
   name: Branch name check
   on: [pull_request]
   jobs:
     check:
       runs-on: ubuntu-latest
       steps:
         - name: Validate branch name
           run: |
             branch="${GITHUB_HEAD_REF}"
             pattern='^(feature|feat|bugfix|fix|hotfix|release|refactor|perf|docs|test|chore|build|ci|experiment|spike)\/([A-Z]+-\d+-)?[a-z0-9][a-z0-9-]*$'
             if ! [[ "$branch" =~ $pattern ]]; then
               echo "Branch name '$branch' does not match $pattern"
               exit 1
             fi
   ```

3. **保护主干分支**：`main`、`develop`、`release/*`、`support/*` 全部开启 **Protected Branch**，要求：
   - 禁止直接 push，必须通过 PR；
   - 至少 1 名 Reviewer 批准；
   - 必需的 CI 检查全部通过；
   - 禁用 force push 与分支删除。
4. **合并后自动删除分支**：GitHub / GitLab 都支持"PR / MR 合并后自动删除 source branch"，开启即可，避免分支列表越积越多。
5. **定期清理过期分支**：对超过 90 天无更新的非保护分支，CI 中自动归档或删除（提前邮件通知作者）。
6. **配合 Conventional Commits**：分支 `type` 与 commit `type` 概念对齐（`feature` ↔ `feat`、`bugfix` ↔ `fix`），降低成员记忆负担；commit 规范见 [commit规范.md](./commit规范.md)。
7. **PR 标题与分支同步校验**：使用 [action-semantic-pull-request](https://github.com/amannn/action-semantic-pull-request) 校验 PR 标题；squash merge 时 PR 标题会成为最终 commit message。
8. **PR 模板里要求关联工单**：在 `.github/pull_request_template.md` 中加入 `Closes #` / `Refs JIRA-XXX` 占位符，强制 PR 关联工单 / Issue。
