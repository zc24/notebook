# Go 结构体规范

> 结构体是 Go 中表达"数据 + 行为"的核心单元。本文整理结构体在**字段设计**与**方法组织**两个维度的常见准则，包括嵌入（embedding）的取舍、字段排列约定，以及结构体方法在文件中的分组顺序。

## 一、结构体中的嵌入

### 1. 嵌入字段放在字段列表顶部

嵌入式类型（如 `mutex`、`io.WriteCloser`）应位于结构体字段列表的**顶部**，并用一个空行与普通字段分隔：

```go
// Bad
type Client struct {
    version int
    http.Client
}
```

```go
// Good
type Client struct {
    http.Client

    version int
}
```

### 2. 嵌入应当切实带来好处

嵌入应当**有意识、有目的地使用**——它会把内嵌类型的导出方法、字段一并暴露到外层，影响外层 API。

> **判断标准**：问自己「是否所有这些导出的内部方法 / 字段都应该直接出现在外层类型上」。如果答案是"部分"或"否"，就**不要嵌入**，改用具名字段。

```go
// Good: Write 委托给内嵌类型，并扩展了计数语义
type countingWriteCloser struct {
    io.WriteCloser
    count int
}

func (w *countingWriteCloser) Write(bs []byte) (int, error) {
    w.count += len(bs)
    return w.WriteCloser.Write(bs)
}
```

嵌入**不应当**：

- 纯粹为了美观或方便。
- 让外层类型更难构造或使用。
- 影响外层类型的零值（如果外层类型本身有有用的零值，嵌入后也应该保持）。
- 作为副作用，把无关的方法或字段公开。
- 公开未导出的类型。
- 影响外层类型的复制语义。
- 改变外层类型的 API 或类型语义。
- 嵌入内部类型的非规范形式。
- 暴露外层类型的实现细节。
- 让用户能观察或控制类型内部状态。
- 通过包装方式改变内部函数的一般行为，给用户带来意外。

### 3. 反例集

```go
// Bad: 直接暴露 Mutex 的 Lock/Unlock 给外部
type A struct {
    sync.Mutex // A.Lock() / A.Unlock() 现在可被外部调用
}
```

```go
// Bad: 用指针/接口嵌入会破坏零值
type Book struct {
    io.ReadWriter // 零值是 nil 接口
    // ...
}

var b Book
b.Read(...)   // panic: nil pointer
b.Write(...)  // panic: nil pointer
```

```go
// Good: 嵌入有有用零值的具体类型
type Book struct {
    bytes.Buffer
    // ...
}

var b Book
b.Read(...)   // ok
b.Write(...)  // ok
```

```go
// Bad: 把多个互不相关的类型一股脑嵌入
type Client struct {
    sync.Mutex
    sync.WaitGroup
    bytes.Buffer
    url.URL
}
```

```go
// Good: 改为具名字段，语义清晰
type Client struct {
    mtx sync.Mutex
    wg  sync.WaitGroup
    buf bytes.Buffer
    url url.URL
}
```

> 即使是未导出类型，**`Mutex` 也不应作为内嵌字段**——零值 `Mutex` 已经是有效的，没必要嵌入；嵌入只会把锁语义暴露到结构体外。

## 二、文件中的函数与方法顺序

为了让阅读者更容易扫到主线代码，**结构体的方法在文件中的排列**（以及与之穿插的普通函数）应满足以下准则：

- **按调用顺序排列**（从被调用方到调用方）。
- **同一文件中的方法按接收者分组**——同一个类型的方法集中在一起，不要被无关函数打散。
- **导出函数优先**：放在 `struct` / `const` / `var` 定义之后、未导出函数之前。
- **构造函数**（`newXYZ` / `NewXYZ`）紧跟其类型定义，位于该类型的其他方法之前。
- **普通工具函数**没有接收者，放在文件末尾即可。

```go
// Bad
func (s *something) Cost() int {
    return calcCost(s.weights)
}

type something struct{ /* ... */ }

func calcCost(n []int) int { /* ... */ }

func (s *something) Stop() { /* ... */ }

func newSomething() *something {
    return &something{}
}
```

```go
// Good
type something struct{ /* ... */ }

func newSomething() *something {
    return &something{}
}

func (s *something) Cost() int {
    return calcCost(s.weights)
}

func (s *something) Stop() { /* ... */ }

func calcCost(n []int) int { /* ... */ }
```

> 这条规则虽然不限于结构体，但对"结构体 + 方法集"的可读性影响尤其显著——按上述顺序排列后，读者打开文件就能从「类型 → 构造函数 → 方法 → 工具函数」一气读完。关于"包内多个文件如何拆分"的更高一层组织，参见 [包与文件组织规范](./包.md)。
