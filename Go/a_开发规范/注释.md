# Go 注释规范

> 注释是代码可读性的最后一道补充，目的是解释「**为什么**」而不是重复「**做了什么**」。本文整理 Go 注释的硬性规则与社区惯例，包括文档注释（Doc Comments）、包注释（Package Comments）、注释句子写法、行长度建议、示例代码、Godoc 格式化语法等。

## 1. 核心原则

- **文档注释优先描述「契约」**：公开的包、类型、函数、方法、常量、变量必须添加文档注释，说明其**目的、行为、入参/返回值约束**。
- **代码本身就是文档**：注释用于补充「为什么这样做」，而不是逐行重复「做了什么」。
- **面向调用方写作**：文档注释最终会出现在 Godoc 与 IDE 提示中，应该写给**任何使用该包的人**看，而不是只写给作者本人看。
- **保持一致**：在同一文件、同一项目内，注释风格（行长、缩进、标点）应保持统一。

> **提示**：开发与代码评审过程中，建议使用 `go doc` 或本地 Godoc 预览注释效果，确认文档与可运行示例呈现符合预期。

## 2. 文档注释（Doc Comments）

### 2.1 哪些声明必须有文档注释

- 所有**顶级导出**（首字母大写）的名称——包、类型、函数、方法、常量、变量——**必须**有文档注释。
- 行为或语义**不明显的未导出**类型/函数声明，也**应当**写注释。
- 文档注释**紧贴**被注释对象，两者**之间不留空行**，否则注释不会被 Godoc 关联到对象上。

### 2.2 注释的基本格式

文档注释应当是**完整的句子**：以被描述对象的**名称开头**、首字母大写、以**句号结尾**。冠词（"a"、"an"、"the"）可以放在名称前面，使句子读起来更自然。

```go
// Good:
// A Request represents a request to run a command.
type Request struct { /* ... */ }

// Encode writes the JSON encoding of req to w.
func Encode(w io.Writer, req *Request) { /* ... */ }
```

```go
// Bad:
// 这是一个请求结构体
type Request struct { /* ... */ }

// 编码并写入
func Encode(w io.Writer, req *Request) { /* ... */ }
```

约定要点：

1. 注释**紧贴**对象，中间不留空行。
2. 第一句**以对象名称开头**（例如 `// Request ...`、`// Encode ...`），这样在 Godoc 索引页与 IDE 提示中能正确显示。
3. 完整句子需要**首字母大写**并以**句号结尾**。
4. 文档注释作用于其**正下方的符号**；若位于结构体内连续多行注释之后跟着一组字段，则应用于这一**字段组**。

### 2.3 未导出声明的文档注释

未导出的代码如果需要写文档注释，应当遵循与导出代码**相同的格式**——同样以名称开头。这样的好处是：将来需要将其导出时，只需替换名称即可，不必重写注释。

```go
// Good:
// parseOptions parses options from a raw string and returns the parsed
// option set, or an error if the input is malformed.
func parseOptions(raw string) (*Options, error) { /* ... */ }
```

### 2.4 结构体字段的注释

结构体字段的注释可以有两种形式：

- **完整文档注释**：放在字段上方，用于字段语义复杂、需要详细解释时。
- **行尾短语注释**：放在字段右侧，适合字段含义简洁、可用短语描述的情况，**省略主语**（默认主语即字段名）。

```go
// Good:
// A Server handles serving quotes from the collected works of Shakespeare.
type Server struct {
    // BaseDir points to the base directory under which Shakespeare's works are stored.
    //
    // The directory structure is expected to be the following:
    //   {BaseDir}/manifest.json
    //   {BaseDir}/{name}/{name}-part{number}.txt
    BaseDir string

    WelcomeMessage  string // displayed when user logs in
    ProtocolVersion string // checked against incoming requests
    PageLength      int    // lines per page when printing (optional; default: 20)
}
```

同一组字段还可以通过**空行分组 + 行内分组标题注释**的方式组织：

```go
// Good:
// Options configure the group management service.
type Options struct {
    // General setup:
    Name  string
    Group *FooGroup

    // Dependencies:
    DB *sql.DB

    // Customization:
    LargeGroupThreshold int // optional; default: 10
    MinimumMembers      int // optional; default: 2
}
```

> **提示**：对于带默认值/可选语义的字段，惯例是在行尾注释中使用 `optional; default: <value>` 这种紧凑写法。

## 3. 包注释（Package Comments）

### 3.1 基本规则

- 包注释**必须紧贴** `package` 子句上方，**之间不能有空行**。
- 每个包**只能有一条**包注释。若包由多个文件组成，**只能在其中一个文件**中写包注释。
- 包注释也必须是完整的英文句子，**首字母大写**并以**句号结尾**。

```go
// Good:
// Package math provides basic constants and mathematical functions.
//
// This package does not guarantee bit-identical results across architectures.
package math
```

### 3.2 main 包的注释

`main` 包的注释格式略有不同：通常以 BUILD 文件中 `go_binary` 规则的名称（即二进制名）作为开头，**取代**包名出现在注释中。

```go
// Good:
// The seed_generator command is a utility that generates a Finch seed file
// from a set of JSON study configs.
package main
```

只要二进制名称与构建文件保持完全一致，下列各种形式都是可接受的：

```go
// Good:
// Binary seed_generator ...
// Command seed_generator ...
// Program seed_generator ...
// The seed_generator command ...
// The seed_generator program ...
// Seed_generator ...
```

> **注意**：注释是公开可见的文档，必须使用正确的英文书写。即使二进制名（如 `seed_generator`）与命令行调用的实际拼写不完全一致，**当它作为句首词时也必须首字母大写**（即 `Seed_generator`）。不允许以小写字母开头的句子作为包注释。

### 3.3 多行包注释

当注释较长、或需要在文档中保留**可被复制粘贴的命令行示例/模板**时，建议使用 `/* ... */` 多行注释，并对代码示例进行**缩进**（Godoc 会据此识别为代码块）。

```go
// Good:
/*
The seed_generator command is a utility that generates a Finch seed file
from a set of JSON study configs.

    seed_generator *.json | base64 > finch-seed.base64
*/
package template
```

### 3.4 doc.go 文件

如果包**没有明显的主文件**，或者包注释**特别长**，可将包注释单独放入名为 `doc.go` 的文件中。该文件应**仅包含**包注释与 `package` 子句，不放任何其他代码。

```go
// doc.go

// Package httpx provides HTTP utilities used across the project.
//
// It includes middleware for logging, recovery, request ID propagation,
// and standardized JSON response helpers.
package httpx
```

### 3.5 文件级维护性注释

那些**面向维护者**（而非外部使用者）且作用于整个文件的注释，应放在 **`import` 声明之后**。这类注释不会出现在 Godoc 中，也不受上述包注释格式规则的约束。

```go
// Good:
package foo

import (
    "fmt"
    "os"
)

// 本文件内的实现依赖于平台特定的系统调用，
// 修改时请同步检查 foo_windows.go 中的对应实现。
```

**落地规则**：

1. **优先**把"面向维护者的整文件注释"放在 `import` 块**之后**，这些注释不会显示在 Godoc 中，也不受上述关于包注释的规则的约束。这是 Google Go Style Decisions 的推荐做法，可以从源头避免与包文档发生歧义。
2. 如果确有理由放在 `package` 子句**上方**，则**必须**与 `package` 行之间**空一行**，否则会被 Godoc 误识别为包文档注释。

```go
// Good: 普通文件注释放在 package 上方时，必须空一行
// Copyright 2025 Example Inc. All rights reserved.
// 本注释为版权声明，不应被 Godoc 收录为包文档。

// Package foo provides ...
package foo
```

```go
// Bad: 非包文档注释与 package 紧贴，会被 Godoc 当作包文档展示
// Copyright 2025 Example Inc. All rights reserved.
// 本注释为版权声明，不应被 Godoc 收录为包文档。
package foo
```

> **原理**：Godoc 判定文档注释的核心规则是「**与声明相邻、中间不能有空行**」（参见 [Go Doc Comments](https://go.dev/doc/comment)、[Effective Go — Commentary](https://go.dev/doc/effective_go#commentary)、[Go Code Review Comments — Package Comments](https://go.dev/wiki/CodeReviewComments#package-comments)）。
> 因此"普通文件注释放在 `package` 上方时要留空行"，本质上是这条相邻规则的**逆向运用**——通过制造空行，让 Godoc 知道这段注释**不是**包文档。

不同场景下的规范要求可以总结如下：

| 场景 | 规范要求 | 出处 |
| --- | --- | --- |
| **包文档注释** | **紧贴** `package`，**不留空行** | Effective Go、Go Doc Comments、Code Review Comments、Google Go Style Decisions |
| **面向维护者的整文件注释（推荐位置）** | 放在 `import` 块**之后** | Google Go Style Decisions §Package comments |
| **普通文件注释放在 `package` 上方（少用）** | 与 `package` 之间**必须留一行空行** | 由 Godoc 相邻规则反推（Go Doc Comments / Effective Go / Code Review Comments） |

## 4. 注释句子（Comment Sentences）

注释根据其完整性，分为两类，遵循不同的标点/大小写规则：

| 类型              | 是否首字母大写 | 是否以句号结尾 | 使用场景                           |
| ----------------- | -------------- | -------------- | ---------------------------------- |
| **完整句子**      | ✅ 是          | ✅ 是          | 文档注释、解释逻辑的注释段落       |
| **短语/句子片段** | ❌ 否          | ❌ 否          | 结构体字段行尾注释、简短的内联标注 |

```go
// Good:
// CalculateTax returns the tax amount for the given purchase. It panics if
// the rate is negative.
func CalculateTax(price float64, rate float64) float64 { /* ... */ }

type Order struct {
    ID       string // unique identifier; assigned by the server
    Amount   int64  // in cents
    Currency string // ISO 4217 currency code
}
```

> **例外**：如果以**未导出的标识符**开头能够让句子更清晰，也可以**以小写开头**——这种写法最好只出现在段落开头。

## 5. 注释行长度（Comment Line Length）

- Godoc **没有固定的行长度**要求。
- 对于较长的注释，应当**手动换行**，避免在不会自动折行的工具中出现超长行。换行参考列宽通常取 **80 或 100 列**——但这**不是硬性规定**。
- **强制换行可能破坏完整性**（例如长 URL、命令行示例），此时允许超长。
- **在同一文件内保持一致**比追求统一列宽更重要。

```go
// Good:
// This is a comment paragraph.
// The length of individual lines doesn't matter in Godoc;
// but the choice of wrapping makes it easy to read on narrow screens.
//
// Don't worry too much about the long URL:
// https://supercalifragilisticexpialidocious.example.com:8080/Animalia/Chordata/Mammalia/Rodentia/Geomyoidea/Geomyidae/
//
// Similarly, if you have other information that is made awkward
// by too many line breaks, use your judgment and include a long line
// if it helps rather than hinders.
```

应避免**将大段文字挤在一行**，这会严重影响阅读体验：

```go
// Bad:
// This is a comment paragraph. While some code editors and viewers will wrap the paragraph for the reader, others will display a very long line that will overflow most windows and require users to scroll horizontally. In addition, even on a screen capable of displaying the entire line, it is easier to read a narrower paragraph than very wide one.
```

## 6. 示例（Examples）

### 6.1 可运行的示例函数

包**应当**通过示例清晰地说明其预期用途。**优先**提供「可运行的示例（runnable examples）」——这些示例会在 Godoc 中渲染，并被 `go test` 实际执行。

约定：

- 可运行示例必须放在**测试文件**（`*_test.go`）中，**不能**放在生产源码文件里。
- 函数名以 `Example` 开头：
  - `ExampleFuncName()` —— 针对 `FuncName` 的示例
  - `ExampleType_Method()` —— 针对 `Type.Method` 的示例
  - `ExampleFuncName_suffix()` —— 同一目标的多个示例用小写后缀区分
- 通过 `// Output:` 注释指定期望输出，`go test` 会校验。

```go
// mathx.go
package mathx

func Abs(x int) int {
    if x < 0 {
        return -x
    }
    return x
}
```

```go
// mathx_example_test.go
package mathx_test

import (
    "fmt"

    "example.com/mathx"
)

func ExampleAbs() {
    fmt.Println(mathx.Abs(-42))
    // Output: 42
}

func ExampleAbs_zero() {
    fmt.Println(mathx.Abs(0))
    // Output: 0
}
```

### 6.2 注释中的代码示例

当**无法**提供可运行示例时（例如示例涉及外部资源、需要复杂的环境准备），可以在文档注释中嵌入代码示例。

约定：

- 嵌入的代码同样需要遵循**标准格式规范**（gofmt 风格）。
- Godoc 通过**缩进**识别代码块——注释中的代码示例行需要相对正文**额外缩进**。

```go
// Good:
// NewClient returns a Client configured against the production endpoint.
//
// Typical usage:
//
//     c, err := pkg.NewClient(ctx)
//     if err != nil {
//         return err
//     }
//     defer c.Close()
//
// The returned Client is safe for concurrent use by multiple goroutines.
func NewClient(ctx context.Context) (*Client, error) { /* ... */ }
```

## 7. 常见反例与最佳实践

| 反例                                                | 推荐写法                                                           |
| --------------------------------------------------- | ------------------------------------------------------------------ |
| 注释与对象之间**留空行**，导致 Godoc 无法关联       | 注释**紧贴**对象，中间不留空行                                     |
| 注释**重复**变量/函数名字面意思（`// i 是索引`）    | 写「为什么」或描述**契约**（前置条件、副作用、并发安全等）         |
| 文档注释**不以对象名开头**                          | 第一句以对象名作主语：`// Encode writes ...`                       |
| 文档注释**用中文 + 没有句号**                       | 文档注释使用**完整英文句子**，首字母大写、以句号结尾               |
| 包内出现**多条包注释**或注释离 `package` 子句有空行 | 全包**只有一条**包注释，**紧贴** `package` 行                      |
| `main` 包注释以包名 `main` 开头                     | 以**二进制名**开头（如 `// The seed_generator command ...`）       |
| 注释挤在一行超长，难以阅读                          | 在 80 或 100 列附近**手动换行**，URL/示例可超长                    |
| 在源码文件内塞入运行示例                            | 将可运行示例放入 `*_test.go`，以 `Example` 开头并使用 `// Output:` |
| 注释中嵌入代码却**未缩进**                          | 嵌入的代码示例必须**缩进**，否则 Godoc 不会识别为代码块            |

> **最佳实践**：对**未导出**代码也按导出代码的注释格式来写（以名称开头）。将来若需要导出，仅需替换名称——既不必重写注释，又能强迫自己在私有 API 上也保持清晰思考。

## 8. Godoc 格式化语法

Godoc 解析注释时使用一套**轻量的格式约定**：理解这些规则后，写出的注释就能在 `go doc`、`pkg.go.dev` 与 IDE 提示中呈现出**清晰的结构**（段落、代码块、标题、锚点），而无需任何额外标记语言。

### 8.1 段落分隔

段落之间必须用**仅有 `//` 的空注释行**分隔。否则 Godoc 会把多行合并为一个段落渲染。

```go
// Good:
// LoadConfig reads a configuration out of the named file.
//
// See some/shortlink for config file format details.
```

### 8.2 缩进代码块（preformatted）

文档注释中，**相对正文额外缩进**的行会被 Godoc 渲染为**原样显示**（preformatted）的代码块：保留空格、字体等宽、不进行自动折行。

```go
// Good:
// Update runs the function in an atomic transaction.
//
// This is typically used with an anonymous TransactionFunc:
//
//   if err := db.Update(func(state *State) { state.Foo = bar }); err != nil {
//     //...
//   }
```

这种「逐字复制」（verbatim）的写法还可以用来呈现 Godoc 本身不支持的结构——例如**列表**、**表格**、**键值对说明**等：

```go
// Good:
// LoadConfig reads a configuration out of the named file.
//
// LoadConfig treats the following keys in special ways:
//   "import" will make this configuration inherit from the named file.
//   "env" if present will be populated with the system environment.
```

> **提示**：能写成可运行 `Example` 函数（见 [§6.1](#61-可运行的示例函数)）就**优先**写成 Example。注释中嵌入的代码示例只是文本，无法被 `go test` 校验，更容易随代码演进而失修。

### 8.3 标题语法

满足下列**全部条件**的单独一行注释，会被 Godoc 识别为**标题**，并自动生成可链接的锚点：

1. **以大写字母开头**；
2. 除 `(`、`)`、`,` 之外**不包含任何标点符号**；
3. 其上下**各跟一个空注释行**（即前后均为空白段落，构成独立的「单行段落」）。

```go
// Good:
// The following line is formatted as a heading.
//
// Using headings
//
// Headings come with autogenerated anchor tags for easy linking.
```

### 8.4 可运行示例与 Godoc 文档联动

测试文件中的 `Example*` 函数会**自动被 Godoc 收录**，并展示在对应符号的文档下方——这是让示例代码**既能跑得通、又能展示给用户**的最佳方式。

```go
// Good:
func ExampleConfig_WriteTo() {
    cfg := &Config{
        Name: "example",
    }
    if err := cfg.WriteTo(os.Stdout); err != nil {
        log.Exitf("Failed to write config: %s", err)
    }
    // Output:
    // {
    //   "name": "example"
    // }
}
```

关于 `Example` 函数的**命名约定**与 `// Output:` 校验机制，详见 [§6.1 可运行的示例函数](#61-可运行的示例函数)。
