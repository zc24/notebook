# Go 注释与可读性规范

> 注释是代码可读性的最后一道补充，目的是解释「为什么」而不是「做了什么」。本文整理公共元素的文档注释要求、命名清晰度建议，以及常见反例。

## 一、核心原则

- **文档注释**：公共的包、类型、函数、常量必须添加 `// Comment` 注释，说明其目的、行为以及参数/返回值。
- **清晰命名**：变量、函数、包名应具有明确含义，避免缩写（除非是广为人知的，例如 `ctx`、`url`、`id`）。
- **代码表达意图**：代码本身应足够清晰；注释用于补充「为什么这样做」，而不是重复「做了什么」。

## 二、为公开元素添加文档注释

```go
// 推荐：为公开元素添加详细注释
// ConfigReader is responsible for loading configuration from a file.
// It supports both JSON and YAML formats.
type ConfigReader struct {
    Path string // Path to the config file.
}

// Load reads the configuration from the file specified in Path.
// Returns an error if the file cannot be read or parsed.
func (c *ConfigReader) Load() error {
    // 实现细节...
    return nil
}
```

约定：

- 文档注释紧贴被注释对象，**之间不留空行**。
- 注释的第一句以被注释对象的名字开头，例如 `// ConfigReader is ...`、`// Load reads ...`。

## 三、清晰命名，避免模糊缩写

```go
// 不推荐：命名模糊
var d int     // 什么的时间？天？duration？
var f func()  // 什么函数？

// 推荐：命名明确，见名知意
var timeoutDuration time.Duration
var onSuccessCallback func()
```
