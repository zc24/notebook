## 一、核心原则

- **显式处理**：不能用 `_` 静默丢弃错误，必须返回、记录或妥善处理。
- **信息具体**：错误信息要包含上下文（哪个操作、哪个参数失败），便于排查。
- **保留链路**：跨层向上传递错误时使用 `fmt.Errorf("...: %w", err)` 进行包装，保留原始错误。
- **就近处理**：能在当前函数处理的错误就地处理，无法处理再向上抛。
- **只记录一次**：同一个错误不要在每一层都打日志，原则上「要么记录，要么返回」二选一（详见 [§6.2 不要重复记录错误](#2-不要重复记录错误)）。
- **分层职责**：dao 抛出错误、service 包装错误、handler 记录错误（详见 [§6.1 分层职责](#1-分层职责dao--service--handler)）。

## 二、不要忽略错误

```go
// 不推荐：忽略错误，调用者无法感知失败
func readFile(filename string) string {
    data, _ := ioutil.ReadFile(filename)
    return string(data)
}

// 推荐：返回错误，由调用方决定如何处理
func readFile(filename string) (string, error) {
    data, err := ioutil.ReadFile(filename)
    if err != nil {
        return "", fmt.Errorf("read file %s: %w", filename, err)
    }
    return string(data), nil
}
```

## 三、提供具体的错误信息

错误信息要包含足够的上下文（操作对象、关键参数），便于日志检索与排障。模糊的 "connection refused" 经过包装应该变成 "call service foo: connection refused" 这样有定位价值的消息。

```go
// 不推荐：错误信息空泛，难以定位
return errors.New("failed")

// 推荐：包含操作对象与上下文
return fmt.Errorf("parse config %q: %w", path, err)
```

同时要避免两类常见反模式。

### 1. 避免 "failed to ..." 这种废话前缀

> 来源：[Uber § Error Wrapping](https://github.com/uber-go/guide/blob/master/style.md#error-wrapping) 明文要求。

`failed to` 前缀在调用栈逐层包装时会像滚雪球一样堆积，最终拼出毫无信息量的长句。

```go
// 不推荐
return fmt.Errorf("failed to create new store: %w", err)

// 推荐
return fmt.Errorf("new store: %w", err)
```

层层包装后的对比：

```text
失控：failed to x: failed to y: failed to create new store: the error
清爽：x: y: new store: the error
```

> 例外：错误**离开本进程**进入另一个系统时（例如写入日志、上报监控、写入消息体），应该明确告知这是一个错误——通常由日志框架的 `err` 字段、字段名前缀（如 `Failed`）或日志级别承担，而不是把 "failed to" 写进错误消息本身。

### 2. 避免重复底层错误已有的信息

> 来源：[Google § Adding information to errors](https://google.github.io/styleguide/go/best-practices.html#adding-information-to-errors) 明文要求。

例如 `os` 包返回的错误本身就带文件路径，再在上层重复写一遍只会让消息更长、没有任何新增信息。

```go
// 不推荐：底层 err 已含文件名，又在上层重复写一遍
if err := os.Open("settings.txt"); err != nil {
    return fmt.Errorf("could not open settings.txt: %v", err)
}

// 推荐：只补充业务语义（为什么这个文件重要）
if err := os.Open("settings.txt"); err != nil {
    return fmt.Errorf("launch codes unavailable: %v", err)
}
```

## 四、不要返回具体 error 类型

> 来源：[Google Go Style Guide § Returning errors](https://google.github.io/styleguide/go/decisions.html#returning-errors) 明文要求。

导出函数即使内部返回的是自定义错误类型（如 `*os.PathError`），**签名上也应写 `error`**：

```go
// 不推荐：签名暴露具体类型
func Bad() *os.PathError { /* ... */ }

// 推荐：签名用 error，文档里说明"实际返回 *PathError"
func Chdir(dir string) error { /* ... */ }
```

原因：Go 的 **nil interface 陷阱**——一个**具体类型的 nil 指针**被包装进 `error` interface 后，interface 本身**不是 nil**，会导致 `if err != nil` 出现意料之外的行为。

```go
type MyError struct{}

func (*MyError) Error() string { return "boom" }

func find() *MyError {
    return nil // 具体类型的 nil 指针
}

func wrap() error {
    return find() // 包装进 interface 后，error 不再是 nil
}

if err := wrap(); err != nil {
    // 永远进入这里，但实际上 find() 并没有出错
}
```

详见 [Go FAQ: Why is my nil error value not equal to nil?](https://go.dev/doc/faq#nil_error)。

## 五、错误包装与判定

### 1. 错误包装的三种选择：原样返回 / `%w` / `%v`

> 来源：[Uber § Error Wrapping](https://github.com/uber-go/guide/blob/master/style.md#error-wrapping) 与 [Google § Placement of %w in errors](https://google.github.io/styleguide/go/best-practices.html#placement-of-w-in-errors) 均有明文。

调用其它函数遇到错误时，有三种处理方式，按"是否需要补上下文、是否允许调用方追溯底层错误"选择：


| 写法                           | 适用场景                                            | 备注                              |
| ---------------------------- | ----------------------------------------------- | ------------------------------- |
| `return err` 原样返回            | 底层错误已经足够定位，且本层没有额外上下文可加                         | 保留原始类型与消息，不会引入"噪音上下文"           |
| `fmt.Errorf("...: %w", err)` | 需要补上下文，且**允许**调用方用 `errors.Is` / `errors.As` 追溯 | **会成为函数契约的一部分**，导出函数最好在文档/测试里说明 |
| `fmt.Errorf("...: %v", err)` | 需要补上下文，但**不希望**调用方依赖底层错误类型                      | 调用方无法匹配底层错误，未来如需开放可平滑切换为 `%w`   |


简而言之：**没新信息就原样抛；要加上下文就在 `%w` / `%v` 间二选一**。

### 2. `%w` 的位置

**包装 sentinel error**（sentinel 用作错误"类别"信号）时，把 `%w` 放在错误字符串**最前面**，让分类信息第一时间可见：

```go
var ErrParse = errors.New("parse error")

// 推荐：分类信号在前
return fmt.Errorf("%w: invalid character in header: %v", ErrParse, err)

// 不推荐：分类信号被埋在末尾
return fmt.Errorf("invalid character in header: %v: %w", err, ErrParse)
```

**包装上游普通错误**时，保持自然语序，把上下文写在前、`%w` 写在后即可：

```go
return fmt.Errorf("parse config %q: %w", path, err)
```

### 3. `errors.Is` / `errors.As` 判定

使用 `%w` 包装错误后，调用方可用 `errors.Is` / `errors.As` 判定具体错误类型：

```go
data, err := readFile("config.yaml")
if err != nil {
    if errors.Is(err, os.ErrNotExist) {
        // 文件不存在，走默认配置
        return defaultConfig(), nil
    }
    return nil, err
}
```

要从错误链里提取自定义类型的字段，则用 `errors.As`：

```go
var pathErr *os.PathError
if errors.As(err, &pathErr) {
    log.Printf("path=%s op=%s", pathErr.Path, pathErr.Op)
}
```

### 4. 不要用字符串匹配判定错误

错误字符串可能随时被改写，靠 `strings.Contains` / 正则匹配判错既脆弱又违反 API 契约。应通过 sentinel 变量或自定义类型判定。

```go
// 不推荐：脆弱的字符串匹配
if strings.Contains(err.Error(), "not found") { /* ... */ }

// 推荐：通过 sentinel
if errors.Is(err, ErrNotFound) { /* ... */ }
```

### 5. 动态错误：`fmt.Errorf` 与自定义错误类型的取舍

对于带动态参数（文件名、ID 等）的错误：

- 调用方**不需要**程序化匹配 → 用 `fmt.Errorf` 即可；
- 调用方**需要**匹配并提取字段 → 定义自定义错误类型（`XxxError`），让调用方通过 `errors.As` 取字段。

> **注意**：一旦导出错误变量（`ErrXxx`）或错误类型（`XxxError`），它们就成为包公共 API 的一部分，后续修改会破坏调用方。能不导出就不导出。

**调用方不需要匹配**：

```go
// package foo
func Open(file string) error {
    return fmt.Errorf("file %q not found", file)
}

// package bar
if err := foo.Open("testfile.txt"); err != nil {
    // 无法对错误做精细化处理
    panic("unknown error")
}
```

**调用方需要匹配并使用字段**：

```go
// package foo
type NotFoundError struct {
    File string
}

func (e *NotFoundError) Error() string {
    return fmt.Sprintf("file %q not found", e.File)
}

func Open(file string) error {
    return &NotFoundError{File: file}
}

// package bar
if err := foo.Open("testfile.txt"); err != nil {
    var notFound *NotFoundError
    if errors.As(err, &notFound) {
        // 拿到 notFound.File 后做针对性处理
    } else {
        panic("unknown error")
    }
}
```

## 六、错误的分层处理与记录

### 1. 分层职责：dao / service / handler

错误不应该让每一层都"啥都做一遍"，要为每一层划清边界：


| 层                 | 职责                     | 典型动作                                                                         |
| ----------------- | ---------------------- | ---------------------------------------------------------------------------- |
| **dao / 基础设施层**   | 抛出基础设施错误               | `return fmt.Errorf("query user: %w", err)`                                   |
| **service / 领域层** | 包装为领域错误，必要时映射 sentinel | `return fmt.Errorf("load user profile: %w", err)` 或 `return ErrUserNotFound` |
| **handler / 控制层** | 统一记录日志、转换成 HTTP/RPC 响应 | `log.Errorw(...)` + `http.Error(w, ..., 500)`                                |


简记：**底层抛出 → 中层包装 → 上层记录**。

### 2. 不要重复记录错误

每一层都对同一个错误 `log.Errorf(...)` 一遍，会带来：

- 日志被同一错误刷屏，掩盖真正有用的上下文；
- 排障时难以分辨「这是 N 个错误」还是「一个错误被打了 N 次」；
- 关键字段（如 traceID、参数）反而被淹没。

**原则**：一条错误链上**只在边界打一次日志**，其余层级只做包装与返回。

- **最外层 / 边界层**（HTTP handler、消息消费入口、定时任务入口）：决定记录日志、返回给客户端或转交监控。
- **中间层**：只做 `fmt.Errorf("...: %w", err)` 包装上下文，**不要打日志**。

```go
// 不推荐：每一层都打一遍日志
func repo() error {
    if err := db.Query(); err != nil {
        log.Errorf("db query failed: %v", err) // 第 1 次
        return err
    }
    return nil
}

func service() error {
    if err := repo(); err != nil {
        log.Errorf("repo failed: %v", err) // 第 2 次（重复）
        return err
    }
    return nil
}

func handler(w http.ResponseWriter, r *http.Request) {
    if err := service(); err != nil {
        log.Errorf("service failed: %v", err) // 第 3 次（重复）
        http.Error(w, "internal error", 500)
        return
    }
}

// 推荐：中间层只包装，边界层统一记录
func repo() error {
    if err := db.Query(); err != nil {
        return fmt.Errorf("query user: %w", err)
    }
    return nil
}

func service() error {
    if err := repo(); err != nil {
        return fmt.Errorf("load user profile: %w", err)
    }
    return nil
}

func handler(w http.ResponseWriter, r *http.Request) {
    if err := service(); err != nil {
        // 唯一一次记录，错误链通过 %w 已经完整保留
        log.Errorw("handle request failed",
            "trace_id", r.Header.Get("X-Trace-Id"),
            "err", err,
        )
        http.Error(w, "internal error", 500)
        return
    }
}
```

> 例外：跨进程/跨异步边界（如发往消息队列前、`defer recover` 中）可以再记录一次，因为错误链已无法靠 `%w` 透传。

### 3. 一次处理：错误的几种正确处置方式

调用方拿到错误后，应当根据契约**只处理一次**，避免「先打日志又把 err 原样返回」这种重复处理。常见的几种正确姿势：

#### (a) 包装错误并返回（最常用，把"如何处理"交给更上层）

```go
u, err := getUser(id)
if err != nil {
    return fmt.Errorf("get user %q: %w", id, err)
}
```

#### (b) 记录错误并优雅降级（操作不是关键路径，失败不应中断主流程）

```go
if err := emitMetrics(); err != nil {
    // 打点失败不应该影响业务
    log.Printf("emit metrics: %v", err)
}
```

#### (c) 匹配特定错误并降级，其它情况照常返回

```go
tz, err := getUserTimeZone(id)
if err != nil {
    if errors.Is(err, ErrUserNotFound) {
        tz = time.UTC // 找不到用户时退化为 UTC
    } else {
        return fmt.Errorf("get user %q: %w", id, err)
    }
}
```

#### (d) 反例：记录后又把原 err 返回

```go
u, err := getUser(id)
if err != nil {
    // BAD：上层很可能也会记一遍，造成日志刷屏
    log.Printf("could not get user %q: %v", id, err)
    return err
}
```

## 七、正反例对比


| 反例                                                                    | 原因                             | 推荐做法                                                  |
| --------------------------------------------------------------------- | ------------------------------ | ----------------------------------------------------- |
| `data, _ := ioutil.ReadFile(f)`                                       | 静默丢弃错误，调用方无法感知失败               | `data, err := ioutil.ReadFile(f)` 并显式处理               |
| `return errors.New("failed")`                                         | 错误信息空泛，无法定位                    | `return fmt.Errorf("parse config %q: %w", path, err)` |
| `fmt.Errorf("failed to create new store: %w", err)`                   | `failed to` 是废话前缀，包装时层层堆积      | `fmt.Errorf("new store: %w", err)`                    |
| `fmt.Errorf("could not open settings.txt: %v", err)`（底层 `err` 已含文件名）  | 重复了底层错误已经提供的信息                 | `fmt.Errorf("launch codes unavailable: %v", err)`     |
| `func Open() *PathError`                                              | 导出函数返回具体类型，触发 nil interface 陷阱 | `func Open() error`（文档里说明实际类型）                        |
| `fmt.Errorf("invalid header: %w", ErrParse)`（`ErrParse` 是分类 sentinel） | 分类信号被埋在字符串后半段，可读性差             | `fmt.Errorf("%w: invalid header", ErrParse)`          |
| `if strings.Contains(err.Error(), "not found") { ... }`               | 通过字符串匹配区分错误，脆弱且违反契约            | `if errors.Is(err, ErrNotFound) { ... }`              |
| 每一层都 `log.Errorf(err)` 后再 `return err`                                | 同一错误被打多遍，刷屏掩盖上下文               | 中间层只 `%w` 包装；仅在边界层记录一次                                |
| 调用方需要按 file 字段判错，被调方返回 `fmt.Errorf("file %q not found", f)`           | 字符串无法被结构化匹配                    | 定义 `NotFoundError{File}`，调用方用 `errors.As` 取字段         |
