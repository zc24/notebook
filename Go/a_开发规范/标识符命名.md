## 1. 标识符命名规则

标识符只能包含 **Unicode 字母、数字和下划线**，且**不能以数字开头**。同时，不能使用以下 Go 关键字作为标识符：


| `break`    | `default`     | `func`   | `interface` | `select` |
| ---------- | ------------- | -------- | ----------- | -------- |
| `case`     | `defer`       | `go`     | `map`       | `struct` |
| `chan`     | `else`        | `goto`   | `package`   | `switch` |
| `const`    | `fallthrough` | `if`     | `range`     | `type`   |
| `continue` | `for`         | `import` | `return`    | `var`    |


---

简单的哲学
Go 的设计哲学之一就是追求简单，因此在命名上一样秉承着简单的总体原则。要想做好 Go 标识符命名（包括 package 命名)，至少要遵循两个原则：
- 简单且一致
- 利用上下文辅助命名

## 2. 命名风格：camelCase 与 PascalCase

- **非导出标识符**：使用 `camelCase`（小驼峰）
- **导出标识符**：使用 `PascalCase`（大驼峰）

> 不要使用 `snake_case`、`Pascal_Snake_Case`、`SCREAMING_SNAKE_CASE` 或 `ALLUPPERCASE` 等其他命名风格。
> 若变量类型为 bool 类型，则名称应以 Has, Is, Can 或 Allow 开头
---

## 3. 缩写词大小写一致

缩写词（如 `API`、`URL` 或 `HTTP`）在标识符中应保持大小写一致。

- ✅ 推荐：`apiKey` 或 `APIKey`
- ❌ 不推荐：`ApiKey`

这条规则同样适用于 `ID`（作为 *identity* 或 *identifier* 的缩写）——因此应写成 `userID` 而非 `userId`。

---

## 4. 坚持使用 ASCII 字符

虽然 Go 允许使用所有 Unicode 字母，但非 ASCII 字母往往会导致代码可读性变差且输入不便，在实际开发中非常罕见。**除非有极其特殊的应用场景，否则标识符应坚持使用 ASCII 字母。**

- 用 `pi` 代替 `π`
- 用 `beta` 代替 `β`
- 用 `naiveBayes` 代替 `naïveBayes`

---

## 5. 避免与内置类型/函数冲突

为避免误导读者及潜在的 Bug，应避免使用与 Go [内置类型](https://pkg.go.dev/builtin)冲突的标识符。

- ❌ 不要创建名为 `int`、`bool` 或 `any` 的变量
- ❌ 不要使用 `min`、`max`、`len` 或 `clear` 作为函数名

---

## 6. 不要在标识符中混入类型名

通常情况下，不要在标识符中包含类型名。例如，避免使用 `fullNameString`、`scoreInt` 或 `float64Amount`。

> **例外场景**：进行类型转换时，为了区分原始变量和转换后的变量，在标识符中加入类型后缀是常见且可接受的做法。

```go
userID := 42
userIDStr := strconv.Itoa(userID)
```

---

## 7. 避免与标准库包名冲突

尽可能避免使用与标准库包名冲突的标识符。这属于 **"软性"约定**，因为标准库占用了许多优质的名称（如 `json`、`js`、`mail`、`user`、`csv`、`path`、`filepath`、`log`、`regexp`、`time` 和 `url`），有时很难找到完美的替代方案。

但是，**绝对应该避免与当前代码中实际导入并使用的包名产生冲突**。例如，如果代码导入了 `url` 和 `net/mail` 包，那么就不要在代码中使用 `url` 和 `mail` 作为标识符。

### 标识符命名正反例对比


| 错误示例                                         | 原因            | 推荐写法                                               |
| -------------------------------------------- | ------------- | -------------------------------------------------- |
| `order.total := 99.99` `func load-user()`    | 不允许使用标点符号     | `orderTotal := 99.99` `func loadUser()`            |
| `const 3rdParty = "x"` `func 2FactorAuth()`  | 不能以数字开头       | `const thirdParty = "x"` `func twoFactorAuth()`    |
| `max_value := 10` `func Fetch_user()`        | 命名格式不规范       | `maxValue := 10` `func FetchUser()`                |
| `type HttpClient struct{}` `func parseXml()` | 缩写词大小写不一致     | `type HTTPClient struct{}` `func parseXML()`       |
| `func GetSessionId()` `type OrderId string`  | `ID` 应全大写     | `func GetSessionID()` `type OrderID string`        |
| `résuméCount := 2` `const Σ = 100`           | 使用了非 ASCII 字母 | `resumeCount := 2` `const sum = 100`               |
| `func clear()` `int := cache.Internal()`     | 与内置类型或函数冲突    | `func clearQueue()` `data := cache.Internal()`     |
| `intCount := 42` `resultSlice := []int{}`    | 名称中包含类型信息     | `count := 42` `results := []int{}`                 |
| `type json struct{}` `var log = newLogger()` | 与标准库包名冲突      | `type payload struct{}` `var logger = newLogger()` |

## 8. 导出与非导出标识符

Go 语言中的标识符是**大小写敏感**的。例如，`apiKey`、`apikey` 和 `APIKey` 是三个不同的标识符。

### 8.1 导出规则

在 Go 中，标识符的可见性完全由**首字母大小写**决定：


| 首字母                  | 可见性                 | 说明               |
| -------------------- | ------------------- | ---------------- |
| **大写**（`PascalCase`） | **导出**（exported）    | 对声明该标识符的包之外的代码可见 |
| **小写**（`camelCase`）  | **非导出**（unexported） | 仅在声明它的包内部可见      |


> **重要提醒**：首字母的大小写具有**实质性的语义意义**，会直接影响代码库的行为。**不应仅仅因为美观而使用大写字母开头**；只有当你确实需要将标识符导出，供包外代码访问时，才应当使用大写字母。

### 8.2 默认非导出原则

> **建议在编写包时，默认使用非导出标识符。仅在确实有需求时才进行导出。**

通常情况下，**导出的内容越少，在重构包内代码时就越不容易影响到代码库的其他部分**。《程序员修炼之道》（*The Pragmatic Programmer*）中有一句名言，稍作改动以适应 Go 的术语：

> 编写"害羞"的代码 —— 即不向其他包暴露非必要信息，且不依赖其他包具体实现的包。

这种"最小导出面"的设计可以带来：

- **更低的耦合**：外部代码无法依赖你未导出的实现细节
- **更安全的重构**：修改非导出标识符不会破坏下游调用方
- **更清晰的 API**：导出列表本身就是包的对外契约

### 8.3 main 包的特殊性

`main` 包**极少会被其他包导入**，因此其中的标识符通常都应该是**非导出的**（以小写字母开头）。

> **最常见的例外**：需要导出结构体字段，以便让使用反射机制的包能够访问，例如：
>
> - `[encoding/json](https://pkg.go.dev/encoding/json)`
> - `[encoding/gob](https://pkg.go.dev/encoding/gob)`
> - `[github.com/jmoiron/sqlx](https://github.com/jmoiron/sqlx)`

```go
package main

// Bad: main 包内的工具函数没必要导出
func ParseConfig(path string) (*Config, error) { /* ... */ }

// Good: 内部使用的函数保持非导出
func parseConfig(path string) (*config, error) { /* ... */ }

// Good (例外): 结构体字段需要被 json 反射访问，必须导出
type config struct {
    ServerAddr string `json:"server_addr"`
    Timeout    int    `json:"timeout"`
}
```

## 9. 标识符长度与描述性

> **核心原则**：标识符的使用位置距离其声明位置越远，其名称就应该越具描述性。

- **作用域窄**：仅在声明位置附近使用，使用简短甚至单字母名称是非常普遍的（如 `for` 循环、`range` 块或极短的函数）
- **作用域大**：远离声明位置被使用时，应使用能清晰描述其含义的名称

Dave Cheney 在其 [Practical Go](https://dave.cheney.net/practical-go) 演讲中给出了一个很好的例子：

```go
func AverageAge(people []Person) int {
    if len(people) == 0 {
        return 0
    }

    var count, sum int
    for _, p := range people {
        count++
        sum += p.Age
    }

    return sum / count
}
```

### 9.1 例子解读

上例中的 `count` 和 `sum` 仅在 `AverageAge` 函数内部使用，作用域很窄。如果换成 `peopleCount` 和 `agesSum` 这样过于详细的名称，反而会显得冗长、噪音过多，让代码不易扫读。

### 9.2 实践建议


| 场景                          | 命名风格       | 示例                                                   |
| --------------------------- | ---------- | ---------------------------------------------------- |
| **简短的功能函数 / 局部循环变量 / 短作用域** | 单个单词，甚至单字母 | `i`、`p`、`count`、`sum`、`buf`                          |
| **长函数 / 方法体 / 跨多个作用域**      | 使用更具描述性的名称 | `peopleCount`、`activeUsers`、`pendingOrderTotal`      |
| **导出标识符 / 包级变量 / 函数返回值**    | 必须具备清晰的语义  | `MaxRetries`、`CalculateOrderTotal`、`ErrUserNotFound` |


> **判断准则**：让读者**在标识符出现的位置就能立刻明白其含义**。
>
> - 如果标识符在 3~5 行内就能看到声明、用途和销毁，**短名字** 更利于阅读
> - 如果标识符跨越几十行、被多次引用，或对外暴露，**描述性名字** 才能避免读者来回翻找上下文

## 10. 包名规范

包名的硬性规则与标识符一致：可以包含 Unicode 字母、数字和下划线，不能以数字开头，且不能是 Go 关键字。**但在实践中，包命名的约定要严格得多**，通常遵循以下原则：

### 10.1 基本规则

- 只包含**小写 ASCII 字母和数字**
- 包名应当**简短、易读**，并能反映包的内容
- 通常使用**简单的名词**（如 `user`、`order`、`slug`），并**优先采用单数形式**
- 多单词时**直接拼接**，不使用分隔符。例如：`ordermanager` ✅，而非 `orderManager` 或 `order_manager` ❌
- 包名太长时可以使用**缩写**。这在标准库中很常见，例如 `expvar`（而非 `exportedvariables`）和 `strconv`（而非 `stringconversion`）

> **关于"单数 vs 复数"**：Go 官方文档（[Package names](https://go.dev/blog/package-names)、[Effective Go](https://go.dev/doc/effective_go#package-names)）和 [Google Go Style Guide](https://google.github.io/styleguide/go/decisions.html#package-names) 都**没有明文禁止复数包名**，只要求 "simple nouns（简单名词）"。
>
> "优先单数"是社区从官方的另一条建议 —— **Avoid repetition（避免与包内类型名重复）** —— 推导出来的实践经验：使用者最终写的是 `pkg.Type`，所以 `user.New()`、`order.Service` 读起来比 `users.NewUser()`、`orders.Order` 更自然。
>
> 当**单数形式与 Go 关键字或常用内建类型冲突**时，使用复数是惯用做法，标准库本身就是这么做的：`strings`（避开 `string`）、`bytes`（避开 `byte`）、`errors`（避开 `error`）、`types`（避开 `type`）。
> 有些基于标注库扩展的库命名，可以使用x结尾，例如：stringx 
> 有些约定俗成的包命名可以打破标准命名规则，例如：third_party（第三方代码）

### 10.2 需要避免的命名

- 避免与常用标准库包名相同
- 不要以 `.` 或 `_` 开头（对 Go 是"不可见"的，会被 `go build`、`go run`、`go test` 完全忽略）
- 避免使用具有特殊含义的目录名作为包名：`vendor`、`testdata`、`internal`
- 避免**全能型**包名：`common`、`util`、`helpers`、`types`、`interfaces`

> **关于全能型包名**：例如，一个名为 `helpers` 的包是包含校验、格式化还是 SQL 相关的辅助工具？还是兼而有之？仅看名字无法得知。这类名称由于缺乏明确的边界和作用域，容易导致包变成各种杂物的"垃圾场"。进而导致该包被整个项目广泛引用，增加循环引用的风险，且对该包的任何改动都可能波及全局。换句话说，全能型包名会鼓励创建"爆炸半径"过大的包。
>
> 如果你想创建一个 `utils` 或 `helpers` 包，请思考是否能将其内容拆分为更聚焦、命名更清晰的小包。

### 10.3 包命名正反例对比


| 错误示例                                                   | 原因           | 推荐做法                                      |
| ------------------------------------------------------ | ------------ | ----------------------------------------- |
| `package 3rdparty` `package 2fa`                       | 不能以数字开头      | `package thirdparty` `package twofa`      |
| `package OrderManager` `package order_manager`         | 命名大小写/分隔符不规范 | `package ordermanager`                    |
| `package o` `package stuff`                            | 过于模糊，缺乏描述性   | `package order` `package slug`            |
| `package ordermanagementsystem`                        | 太长，难以输入      | `package order` `package ordermgr`        |
| `package users` `package orders`                       | 复数形式与类型名重复（`users.User`），调用方更啰嗦 | `package user` `package order`（除非单数与关键字冲突，如 `strings`/`bytes`/`errors`） |
| `package url` `package mail`                           | 与标准库包名冲突     | `package links` `package mailer`          |
| `package _cache` `package .hidden`                     | 会被 Go 工具链忽略  | `package cache` `package hidden`          |
| `package internal` `package vendor` `package testdata` | Go 中的特殊目录名   | `package internalauth` `package supplier` |
| `package utils` `package helpers`                      | 作用域不明确的全能型名称 | `package validation` `package formatting` `package filex` |

## 11. 文件命名规范

理想情况下，`.go` 文件名应当**简洁地概括文件内容，最好是单个单词且全部小写**。在标准库 `net/http` 包中，`cookie.go`、`server.go` 和 `status.go` 都是非常好的命名示例。

### 11.1 多单词文件命名

如果你无法用一个单词概括，需要使用两个或更多单词时，目前并没有统一的分隔符规范。即便在 Go 标准库内部，命名方式也不尽相同：

- 下划线分隔：`routing_index.go`、`routing_tree.go`
- 直接拼接：`batchcursor.go`、`textreader.go`、`reverseproxy.go`

> **建议**：在同一个项目中统一选择其中一种风格。推荐**直接拼接单词**（如 `routingindex.go`），并将下划线保留给具有"特殊含义的文件名后缀"。

### 11.2 具有特殊含义的文件名

某些文件名前缀和后缀具有特定用途，**除非你需要触发特定行为，否则应避免使用它们**：


| 模式              | 含义                                               |
| --------------- | ------------------------------------------------ |
| 以 `.` 或 `_` 开头  | 对 Go 工具链不可见，被 `go build`、`go run`、`go test` 完全忽略 |
| `_test.go` 结尾   | 仅由 `go test` 执行，`go run`/`go build` 会忽略          |
| `_<os>.go` 后缀   | 仅在为特定操作系统编译时包含                                   |
| `_<arch>.go` 后缀 | 仅在为特定架构编译时包含                                     |


**操作系统后缀**：`_aix.go`、`_android.go`、`_darwin.go`、`_dragonfly.go`、`_freebsd.go`、`_illumos.go`、`_ios.go`、`_js.go`、`_linux.go`、`_netbsd.go`、`_openbsd.go`、`_plan9.go`、`_solaris.go`、`_wasip1.go`、`_windows.go`

**架构后缀**：`_386.go`、`_amd64.go`、`_arm.go`、`_arm64.go`、`_loong64.go`、`_mips.go`、`_mips64.go`、`_mips64le.go`、`_mipsle.go`、`_ppc64.go`、`_ppc64le.go`、`_riscv64.go`、`_s390x.go`、`_wasm.go`

## 12. 避免冗余命名

> **核心原则**：在命名导出函数、类型、方法时，应尽量避免重复其所属包/类型的名称。

### 12.1 函数命名

如果你有一个名为 `customer` 的包，那么 `NewCustomer()` 或 `CustomerOrders()` 这样的函数名就显得过于冗余。在包外调用时：

- ❌ `customer.NewCustomer()` 和 `customer.CustomerOrders()` 重复了 "customer" 这个词
- ✅ `customer.New()` 和 `customer.Orders()` 读起来更自然

### 12.2 类型命名

在 `customer` 包中表示地址或电话号码时：

- ❌ `CustomerAddress`、`CustomerPhoneNumber`
- ✅ `Address`、`PhoneNumber`

### 12.3 方法命名

例如，在为 `Token` 类型编写方法时：

- ❌ `ValidateToken()`、`IsTokenExpired()`
- ✅ `Validate()`、`IsExpired()`

### 12.4 冗余命名正反例


| 错误示例                                                      | 原因     | 推荐做法                                      |
| --------------------------------------------------------- | ------ | ----------------------------------------- |
| `customer.NewCustomer()` `customer.CustomerOrders()`      | 函数调用冗余 | `customer.New()` `customer.Orders()`      |
| `customer.CustomerAddress` `customer.CustomerPhoneNumber` | 类型引用冗余 | `customer.Address` `customer.PhoneNumber` |

## 13. 方法接收者命名

按照惯例，方法接收者（receiver）应使用**短名称，通常为 1 到 3 个字符**，且往往是该类型名称的缩写。

- `Customer` 类型 → 接收者命名为 `c` 或 `cus`
- `HighScore` 类型 → 接收者命名为 `hs`

> [Go 代码审查指南](https://go.dev/wiki/CodeReviewComments#receiver-names)中明确建议**不要使用 `this`、`self` 或 `me`** 等泛化名称作为接收者。

### 一致性原则

同一个类型的所有方法都应使用**相同的接收者名称**，不要在一个方法中使用 `c`，而在另一个方法中使用 `cus`。

```go
type Order struct {
    Items int
}

// Good: 使用简短的接收者
func (o *Order) Validate() bool {
    return o.Items > 0
}

// Bad: 接收者名字过长
func (order *Order) Validate() bool {
    return order.Items > 0
}

// Bad: 使用了泛化名称
func (self *Order) Validate() bool {
    return self.Items > 0
}
```

## 14. 结构体的 Getter 和 Setter

通常情况下，**Go 语言中的结构体不需要专门编写 Getter 和 Setter 方法**。直接访问结构体字段来读取或修改数据即可。

> **唯一的例外**：当结构体包含未导出字段，但又需要允许包外访问或修改该字段值时，需要创建导出的 Getter 和 Setter 方法。

### 命名约定

- **Setter** 方法名应以 `Set` 为前缀
- **Getter** 方法名**不加** `Get` 前缀

```go
type Customer struct {
    address string
}

func (c *Customer) Address() string {
    return c.address
}

func (c *Customer) SetAddress(addr string) {
    c.address = addr
}
```

## 15. 接口命名

根据惯例，**仅包含单个方法的接口**应以方法名加上 `-er` 后缀或类似形式命名。

```go
type Speaker interface {
    Speak() string
}

type Authorizer interface {
    Authorize(ctx context.Context, action string) error
}

type Authenticator interface {
    Authenticate(ctx context.Context) (User, error)
}
```

Go 标准库中有很多遵循此惯例的接口：

- `[io.Reader](https://pkg.go.dev/io#Reader)`
- `[io.Writer](https://pkg.go.dev/io#Writer)`
- `[fmt.Stringer](https://pkg.go.dev/fmt#Stringer)`

当然也有一些例外，比如 `[http.Handler](https://pkg.go.dev/net/http#Handler)`。

> **注意**：避免在名称中包含类型信息的原则同样适用于接口。除非实在找不到更合适的命名，否则不要使用类似 `UserInterface` 或 `OrderInterface` 这样的名称。

## 16. 错误（error）命名

> 本节规则取**三方最大公约数**：[Uber Go Style Guide](https://github.com/uber-go/guide/blob/master/style.md#error-naming)（唯一明文规定 `Err` / `Error` 前后缀的大厂规范）、[Google Go Style Guide](https://google.github.io/styleguide/go/decisions.html#errors) 与 [Google 最佳实践](https://google.github.io/styleguide/go/best-practices.html#error-structure)（示例 100% 遵循同一套约定）、[Go 官方 CodeReviewComments](https://go.dev/wiki/CodeReviewComments#error-strings) 与 [Effective Go](https://go.dev/doc/effective_go#errors)（明文规定错误字符串写法）。
>
> **本节只覆盖错误的「命名」**。错误处理相关规则（不要返回具体 error 类型、错误包装的选择与位置、`failed to` 反模式等）请参考 [错误处理规范](./错误.md)。

### 16.1 Sentinel error 变量使用 Err 前缀

对存储为**全局变量**的 sentinel 错误（用 `errors.New` 或 `fmt.Errorf` 创建、供调用方用 `errors.Is` 匹配的错误值），按是否导出加前缀：

- **导出**：使用 `Err` 前缀
- **非导出**：使用 `err` 前缀

> 来源：[Uber Go Style Guide § Error Naming](https://github.com/uber-go/guide/blob/master/style.md#error-naming) 明文规则。该规则**覆盖** Uber 自己关于"非导出全局变量加 `_` 前缀"的更通用条款——error 不写成 `_errNotFound`。

```go
var (
    // 导出 sentinel：调用方可用 errors.Is(err, foo.ErrBrokenLink) 匹配
    ErrBrokenLink   = errors.New("link is broken")
    ErrCouldNotOpen = errors.New("could not open")

    // 非导出 sentinel：只在包内用 errors.Is(err, errNotFound) 匹配
    errNotFound = errors.New("not found")
)
```

> **提醒**：导出 sentinel 一旦发布就成为包公开 API 的一部分，**改名 / 删除都属于破坏性变更**。命名前务必想清楚是否真的需要导出。

### 16.2 自定义 error 类型使用 Error 后缀

当错误信息包含**动态字段**、且调用方需要用 `errors.As` 提取这些字段时，应定义**自定义 error 类型**，类型名使用 `Error` 后缀：

- **导出**：`PascalCase` + `Error` 后缀
- **非导出**：`camelCase` + `Error` 后缀

> 来源：[Uber Go Style Guide § Error Naming](https://github.com/uber-go/guide/blob/master/style.md#error-naming) 明文规则；[Google 最佳实践 § Error Structure](https://google.github.io/styleguide/go/best-practices.html#error-structure) 全部示例（`MissingPieceError`、`SimulationError`、`PutsSelfIntoCheckError`）皆遵循。

```go
// Good: 导出自定义错误类型
type NotFoundError struct {
    File string
}

func (e *NotFoundError) Error() string {
    return fmt.Sprintf("file %q not found", e.File)
}

// Good: 非导出自定义错误类型
type resolveError struct {
    Path string
}

func (e *resolveError) Error() string {
    return fmt.Sprintf("resolve %q", e.Path)
}
```

调用方按如下方式匹配并提取信息：

```go
if err := foo.Open("testfile.txt"); err != nil {
    var notFound *NotFoundError
    if errors.As(err, &notFound) {
        log.Printf("缺失文件：%s", notFound.File)
    }
}
```

> **选型速查**（来自 Uber 文档原表）：
>
> | 是否需要 `errors.Is`/`As` 匹配？ | 错误信息 | 推荐做法 |
> | --- | --- | --- |
> | 否 | 静态字符串 | `errors.New` |
> | 否 | 动态字符串 | `fmt.Errorf` |
> | 是 | 静态字符串 | 顶层 `var` + `errors.New` → `ErrXxx` |
> | 是 | 动态字符串 | 自定义类型 → `XxxError` |

### 16.3 错误字符串写法

> 来源：[Go 官方 CodeReviewComments § Error Strings](https://go.dev/wiki/CodeReviewComments#error-strings) 与 [Google § Error strings](https://google.github.io/styleguide/go/decisions.html#error-strings) **都明文规定**。

**硬性规则**：

1. **首字母不大写**（除非以专有名词、缩写词或导出标识符开头）
2. **末尾不加标点**（句号、感叹号等）

原因：错误字符串常嵌入到更长的上下文中输出，例如 `log.Printf("Reading %s: %v", filename, err)`，若首字母大写或末尾带句号，拼接出来的句子就会出现"句中突兀的大写字母 / 多余的句号"。

```go
err := fmt.Errorf("Something bad happened.")
err := fmt.Errorf("something bad happened")
```

**软性建议**（来自 [Effective Go](https://go.dev/doc/effective_go#errors)）：错误字符串最好**带上包名 / 操作名作为前缀**，使其离开上下文也能被识别来源。例如标准库 `image` 包返回的解码错误是 `"image: unknown format"`。

**注意**：日志、测试失败消息、API 返回给终端用户的文案**不适用**上面这条规则——它们是面向人类的完整句子，按正常句子大小写即可：

```go
log.Infof("Operation aborted: %v", err)
t.Errorf("Op(%q) failed unexpectedly; err=%v", args, err)
```

### 16.4 标准库中的命名印证

Go 标准库的 error 命名**完全印证**了上面的规则，可作为权威参考。

**`Err` 前缀的 sentinel 变量**：

```go
io.ErrUnexpectedEOF
io.ErrShortWrite
io.ErrClosedPipe
os.ErrNotExist
os.ErrPermission
os.ErrInvalid
fs.ErrNotExist
sql.ErrNoRows
http.ErrServerClosed
http.ErrNoCookie
```

**`Error` 后缀的自定义类型**：

```go
os.PathError
os.LinkError
os.SyscallError
net.OpError
net.AddrError
net.DNSError
json.SyntaxError
json.UnmarshalTypeError
time.ParseError
strconv.NumError
exec.ExitError
```

**少数命名例外**（均为**早于规范成型**的历史命名，新代码请勿模仿）：

| 标识符 | 应有命名 | 例外原因 |
| --- | --- | --- |
| `io.EOF` | `io.ErrEOF` | Go 1.0 之前确立的事实标准 |
| `context.Canceled` | `context.ErrCanceled` | `context` 包 API 设计时刻意选择 |
| `context.DeadlineExceeded` | `context.ErrDeadlineExceeded` | 同上 |
| `url.Error` | `url.URLError` | 直接叫 `Error`，没加前缀（会和包名冗余） |

### 16.5 错误命名正反例对比

| 错误示例 | 原因 | 推荐做法 |
| --- | --- | --- |
| `var NotFound = errors.New("not found")` | 全局 sentinel 变量缺少 `Err` 前缀 | `var ErrNotFound = errors.New("not found")` |
| `var _errNotFound = errors.New("not found")` | 非导出 sentinel 不应套用 `_` 前缀 | `var errNotFound = errors.New("not found")` |
| `type NotFound struct{ ... }`（实现了 `Error()`） | 自定义 error 类型缺少 `Error` 后缀 | `type NotFoundError struct{ ... }` |
| `type ErrorNotFound struct{ ... }` | 自定义类型用了 `Error` 前缀而非后缀 | `type NotFoundError struct{ ... }` |
| `errors.New("Something bad happened.")` | 错误字符串首字母大写 + 句末标点 | `errors.New("something bad happened")` |

> 错误处理相关的反例（`failed to` 前缀、返回具体 error 类型、字符串匹配判错等）请见 [错误处理规范 §七 正反例对比](./错误.md#七正反例对比)。

## 17. 常量命名

常量名应当**描述这个值是什么**（实体的属性），而不是**描述这个值会被怎么使用**（动作、场景、流程）。

判断方法：把常量名翻译成自然语言，应该读作「**XX 的 YY 值**」这种名词短语；如果读起来像「**做某事时用的那个数**」，就说明命名跑偏了。

### 17.1 描述属性而非动作

| 反例 | 问题说明 | 正例 |
| --- | --- | --- |
| `const RetryWhenServerError = 3` | 用「什么时候用」命名，绑定单一场景 | `const MaxRetries = 3` |
| `const TimeBetweenHealthChecks = 5 * time.Second` | 用「两次动作之间」描述，本质是个间隔 | `const HealthCheckInterval = 5 * time.Second` |
| `const UsernameInputValidationLimit = 32` | 用「输入校验」这个动作命名，换个场景就不通 | `const MaxUsernameLength = 32` |
| `const PageSizeForAdminList = 20` | 把「在哪用」塞进了名字 | `const DefaultPageSize = 20` |
