# Go 单元测试规范

> Go 在同一个目录下允许两种测试包共存：与被测代码同包的 **internal 测试**（白盒），以及独立 `_test` 包的 **external 测试**（黑盒）。本文整理两者的差异、各自的适用场景，以及在工程实践中如何组合使用。

## 一、两种测试包的核心区别

Go 工具链允许在同一个目录下，测试文件以两种 `package` 声明并存。两者在编译期会被分别编译，最终都链接进 `go test` 生成的测试二进制中。

| 写法 | 称呼 | 视角 | 访问能力 |
| --- | --- | --- | --- |
| `package foo` | **internal 测试**（白盒） | 实现者视角 | 可访问包内所有 **未导出** 标识符 |
| `package foo_test` | **external 测试**（黑盒） | 使用者视角 | 只能通过 **导出 API** 访问 |

> **Note**：`foo_test` 是 Go 工具链唯一允许与被测包同目录共存的"额外包名"，且只能用于 `_test.go` 文件。普通 `.go` 文件中出现 `package foo_test` 会编译失败。

## 二、Internal 测试：`package foo`

适用场景：**测试对象本身是包的内部细节**。

1. **测试未导出函数 / 方法 / 类型**
   纯工具函数、内部状态机、私有算法等只能从同包访问。

   ```go
   package cache

   func Test_evict(t *testing.T) {
       c := &lru{cap: 2}
       c.put("a", 1)
       c.put("b", 2)
       c.put("c", 3)
       if _, ok := c.get("a"); ok {
           t.Fatal("expect a evicted")
       }
   }
   ```

2. **需要替换 / 打桩包内变量（monkey patch）**
   例如把内部的 `var nowFunc = time.Now` 临时替换成可控时钟，仅同包测试能改。

3. **覆盖率要求高的核心包**
   需要覆盖内部分支与错误路径，而这些路径未通过导出 API 暴露。

4. **小型工具包、纯算法包**
   API 与实现耦合紧密，写黑盒测试反而绕一圈。

> **Tip**：对于多数业务内部包而言，**默认就是 internal 测试**——直接 `package foo`，无需额外考虑。

## 三、External 测试：`package foo_test`

适用场景：**站在使用者角度验证包的对外契约**。

1. **库 / SDK / 公共组件**
   通过 external 测试驱动 API 设计：如果测试都写不出来，说明 API 设计有问题。

2. **`ExampleXxx` 示例函数**
   `go doc` 渲染出来的 example 必须像真实调用方一样 `import "yourpkg"`，所以基本都放在 `foo_test` 包里。这是 Go 官方明确推荐的做法。

   ```go
   package json_test

   import (
       "encoding/json"
       "fmt"
   )

   func ExampleMarshal() {
       b, _ := json.Marshal(map[string]int{"a": 1})
       fmt.Println(string(b))
       // Output: {"a":1}
   }
   ```

3. **避免循环依赖**
   被测包 `foo` 依赖了 `bar`，而测试时又需要 `bar` 的工具——若以 `package foo` 写测试，会形成 `foo -> bar -> foo` 的导入环。换成 `foo_test` 即可解开：`foo_test` 不属于 `foo` 的依赖链。

   标准库的 `net/http`、`encoding/json` 大量使用这一技巧。

4. **集成测试 / 跨包行为测试**
   测试涉及多个包协作，从外部角度验证更合理。

5. **强制只用公开 API**
   防止测试无意间依赖私有实现，让重构更安全。

> **Tip**：把"是否能写出黑盒测试"当作 API 的健壮性检测——这是 external 测试最大的隐性价值。

## 四、混合使用与 `export_test.go` 技巧

**同一个目录下两种包可以共存**，按被测对象分别放置即可：

```text
cache/
├── lru.go
├── lru_test.go          // package cache       — 测内部 evict、桶大小等
├── lru_export_test.go   // package cache       — 通过别名"测试专用导出"
└── example_test.go      // package cache_test  — 测 Get/Put 公共 API、Example
```

如果偶尔需要在黑盒测试里访问一个未导出符号，可以在 **同包测试文件** 中通过 `export_test.go` 显式导出，仅在测试编译期生效：

```go
// lru_export_test.go
package cache

// Evict 仅在 go test 时存在，对外部使用者完全不可见。
var Evict = (*lru).evict
```

然后在 `cache_test` 包里就能 `cache.Evict(...)` 调用：

```go
// example_test.go
package cache_test

import (
    "testing"

    "example.com/cache"
)

func TestEvictFromOutside(t *testing.T) {
    // 通过 export_test.go 暴露的 Evict 间接访问内部逻辑
    _ = cache.Evict
}
```

这是标准库的经典模式，例如 `runtime/export_test.go`、`time/export_test.go`。

> **Note**：`export_test.go` 这个文件名没有任何魔法，关键在 `_test.go` 后缀。叫 `internal_test.go`、`hooks_test.go` 都可以——只要带 `_test.go` 后缀，文件就只在 `go test` 期间参与编译。

## 五、文件命名与组织

### 5.1 权威规范只约束「包名」和「文件后缀」

关于测试文件，**官方与大厂规范只规定了两点**：

- **文件名必须以 `_test.go` 结尾**（Go 工具链强制要求）。来源：[Go 官方 testing 包文档](https://pkg.go.dev/testing)。
- **黑盒测试包名为 `foo_test`，集成测试包名可使用 `xxx_service_test`** 等下划线形式（属于"包名禁用下划线"的例外）。来源：[Google Go Style Guide - Package names](https://github.com/google/styleguide/blob/gh-pages/go/decisions.md#package-names)。

对 external 测试文件**具体叫什么名字**（如要不要加 `_external_test.go` 后缀），Go 官方、Google Go Style Guide、Uber Go Style Guide **均无规定**。

### 5.2 推荐做法：对齐 Go 标准库

标准库的实际做法是 **按"被测对象 / 功能"命名文件，靠 `package` 声明区分内外**，而不是用文件名后缀来区分：

```text
net/http/
├── client.go
├── client_test.go      // package http_test  — 文件名不加后缀，靠 package 区分
├── export_test.go      // package http       — 测试专用导出
└── ...

encoding/json/
├── decode_test.go      // package json       — internal
├── encode_test.go      // package json
└── example_test.go     // package json_test  — Example 集中放置
```

由此抽出几条可落地的命名准则：

- 测试文件与被测文件同名：`client.go` ↔ `client_test.go`。
- `ExampleXxx` 集中放在 `example_test.go`（`package foo_test`），与 godoc 渲染习惯一致。
- 测试专用导出放在 `export_test.go`（`package foo`），命名沿用标准库习惯。
- 同一个被测对象的 internal / external 测试**不能混在一个文件里**——一个文件只能声明一个包名。

### 5.3 可选做法：用文件后缀显式区分内外

如果团队希望在文件名上一眼看出是黑盒还是白盒，可以约定一个本地后缀，例如：

```text
client.go               // package foo        — 实现
client_test.go          // package foo        — 内部白盒测试
client_external_test.go // package foo_test  — 外部黑盒测试
```

> **Note**：`_external_test.go` 这类后缀**不是官方或大厂规范**，只是部分博客与团队的本地约定（例如 Redowan Delowar 的 [Organizing Go tests](https://rednafi.com/go/organizing-tests/) 一文）。标准库本身不使用这种命名。**项目内统一即可，不必跨项目强求**。
>
> **Tip**：测试用例的函数名允许使用下划线分组，如 `TestParser_InvalidInput`。这一点 Google Go Style Guide 与 Go 官方 effective_go 都明确认可。

## 六、决策清单

下面这张表把日常写测试时的取舍浓缩成几条经验法则：

| 场景 | 推荐放置 |
| --- | --- |
| 测试未导出函数 / 方法 / 内部状态 | `package foo` |
| 需要替换包内变量做打桩 | `package foo` |
| 小型工具包、内部业务包（默认） | `package foo` |
| 库 / SDK / 公共组件的对外行为 | `package foo_test` |
| `ExampleXxx` 示例函数 | `package foo_test` |
| 解决测试中出现的 import 循环 | `package foo_test` |
| 跨包集成 / 端到端测试 | `package foo_test` |
| 想强制测试只用公开 API | `package foo_test` |
| 偶尔需要从外部测试访问内部符号 | `package foo` + `export_test.go` 暴露别名 |
