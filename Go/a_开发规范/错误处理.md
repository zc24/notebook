## 一、核心原则

- **显式处理**：不能用 `_` 静默丢弃错误，必须返回、记录或妥善处理。
- **信息具体**：错误信息要包含上下文（哪个操作、哪个参数失败），便于排查。
- **保留链路**：跨层向上传递错误时使用 `fmt.Errorf("...: %w", err)` 或 `errors.Wrap` 进行包装，保留原始错误。
- **就近处理**：能在当前函数处理的错误就地处理，无法处理再向上抛。
- **只记录一次**：同一个错误不要在每一层都打日志，原则上「要么记录，要么返回」二选一（详见 [§10.2 不要重复记录错误](#2-不要重复记录错误)）。
- **分层职责**：dao 抛出错误、service 包装错误、handler 记录错误（详见 [§10.1 分层职责](#1-分层职责dao--service--handler)）。

## 二、不要忽略错误

```go
// 不推荐：忽略错误，调用者无法感知失败
func readFile(filename string) string {
    data, _ := ioutil.ReadFile(filename)
    return string(data)
}

// 推荐：返回错误，由调用方决定如何处理
func readFile(filename string) (string, error) {
    data, err := ioutil.ReadFile(filename)
    if err != nil {
        return "", fmt.Errorf("read file %s: %w", filename, err)
    }
    return string(data), nil
}
```

> 在极少数情况下，如果可以忽略或放弃某个错误的话（比如，对于那些被记录为永远不会出错的 `(*bytes.Buffer).Write` 调用），则应附上注释说明为何这样做是安全的。

```go
// 推荐：明确注释"为何安全"
var b *bytes.Buffer
n, _ := b.Write(p) // never returns a non-nil error
```

## 三、错误消息的写法

### 1. 提供具体的上下文

错误信息要包含足够的上下文（操作对象、关键参数），便于日志检索与排障。模糊的 `connection refused` 经过包装应该变成 `call service foo: connection refused` 这样有定位价值的消息。

```go
// 不推荐：错误信息空泛，难以定位
return errors.New("failed")

// 推荐：包含操作对象与上下文
return fmt.Errorf("parse config %q: %w", path, err)
```

### 2. 避免 "failed to ..." 这种废话前缀

> 来源：[Uber § Error Wrapping](https://github.com/uber-go/guide/blob/master/style.md#error-wrapping) 明文要求。

`failed to` 前缀在调用栈逐层包装时会像滚雪球一样堆积，最终拼出毫无信息量的长句。

```go
// 不推荐
return fmt.Errorf("failed to create new store: %w", err)

// 推荐
return fmt.Errorf("new store: %w", err)
```

层层包装后的对比：

```text
失控：failed to x: failed to y: failed to create new store: the error
清爽：x: y: new store: the error
```

> 例外：错误**离开本进程**进入另一个系统时（例如写入日志、上报监控、写入消息体），应该明确告知这是一个错误——通常由日志框架的 `err` 字段、字段名前缀（如 `Failed`）或日志级别承担，而不是把 `failed to` 写进错误消息本身。

### 3. 避免重复底层错误已有的信息

> 来源：[Google § Adding information to errors](https://google.github.io/styleguide/go/best-practices.html#adding-information-to-errors) 明文要求。

例如 `os` 包返回的错误本身就带文件路径，再在上层重复写一遍只会让消息更长、没有任何新增信息。

```go
// 不推荐：底层 err 已含文件名，又在上层重复写一遍
if err := os.Open("settings.txt"); err != nil {
    return fmt.Errorf("could not open settings.txt: %v", err)
}

// 推荐：只补充业务语义（为什么这个文件重要）
if err := os.Open("settings.txt"); err != nil {
    return fmt.Errorf("launch codes unavailable: %v", err)
}
```

### 4. 小写开头、无结尾标点

> 来源：[Google § Error strings](https://google.github.io/styleguide/go/decisions.html#error-strings) 明文要求。

`error.Error()` 返回的字符串通常会被嵌入到其它消息里（如 `cannot start server: %v`），因此：

- **首字母小写**（除非以导出名、专有名词或缩写开头）；
- **结尾不带标点**（句号、感叹号等）。

```go
// 不推荐
err := fmt.Errorf("Something bad happened.")

// 推荐
err := fmt.Errorf("something bad happened")
```

而**完整展示给用户的消息**（日志、测试失败、API 响应、用户界面文案）则**首字母通常大写**，这两套规则不要混淆：

```go
// 推荐：作为完整消息展示，首字母大写
log.Infof("Operation aborted: %v", err)
log.Errorf("Operation aborted: %v", err)
t.Errorf("Op(%q) failed unexpectedly; err=%v", args, err)
```

## 四、错误返回值的惯用法

### 1. `error` 作为最后一个返回值

按 Go 惯例，凡是可能失败的函数都用 `error` 表示失败；`error` **必须是最后一个返回值**。

```go
// 推荐
func Good() error { /* ... */ }

func GoodLookup() (*Result, error) {
    // ...
    if err != nil {
        return nil, err
    }
    return res, nil
}
```

返回 `nil` 错误是表示"本来可能失败、实际成功"的惯用方式。函数返回非 `nil` 错误时，调用方应将**所有非错误返回值视为未指定**——除非函数文档明确写明在出错时仍有意义。通常情况下出错时其它返回值为零值，但不可一概而论。

### 2. 不要返回具体 error 类型

> 来源：[Google § Returning errors](https://google.github.io/styleguide/go/decisions.html#returning-errors) 明文要求。

导出函数即使内部返回的是自定义错误类型（如 `*os.PathError`），**签名上也应写 `error`**：

```go
// 不推荐：签名暴露具体类型
func Bad() *os.PathError { /* ... */ }

// 推荐：签名用 error，文档里说明"实际返回 *PathError"
func Chdir(dir string) error { /* ... */ }
```

原因：Go 的 **nil interface 陷阱**——一个**具体类型的 nil 指针**被包装进 `error` interface 后，interface 本身**不是 nil**，会导致 `if err != nil` 出现意料之外的行为。

```go
type MyError struct{}

func (*MyError) Error() string { return "boom" }

func find() *MyError {
    return nil // 具体类型的 nil 指针
}

func wrap() error {
    return find() // 包装进 interface 后，error 不再是 nil
}

if err := wrap(); err != nil {
    // 永远进入这里，但实际上 find() 并没有出错
}
```

详见 [Go FAQ: Why is my nil error value not equal to nil?](https://go.dev/doc/faq#nil_error)。

### 3. 接受 `context.Context` 的函数应返回 `error`

接受 `context.Context` 参数的函数通常涉及阻塞 / IO / 异步等待，调用方需要能感知"上下文被取消"。**这类函数应当返回 `error`**，否则调用方无法判别 `ctx.Done()` 触发后的状态。

```go
// 推荐：context.Context 与 error 成对出现
func Fetch(ctx context.Context, id string) (*User, error) { /* ... */ }

// 不推荐：吃掉 ctx 但不返回 error，调用方无法感知取消
func Fetch(ctx context.Context, id string) *User { /* ... */ }
```

## 五、用多返回值代替带内错误

> 来源：[Google § In-band errors](https://google.github.io/styleguide/go/decisions.html#in-band-errors) 明文要求。

在 C 等语言中，函数常用 `-1`、`null`、空字符串等"带内值"表示失败。这种「带内错误（in-band errors）」做法在 Go 中应该避免——一旦调用方忘了判，错误会被悄悄传递下去，最终归咎到根本无辜的函数头上。

```go
// 不推荐：用 -1 表示"没找到"
// Lookup returns the value for key or -1 if there is no mapping for key.
func Lookup(key string) int

// 调用方忘记判 -1 时，Parse 会被错误地归咎为出错方
return Parse(Lookup(missingKey))
```

Go 支持多返回值，应额外返回 `error` 或 `bool` 来表示"其它返回值是否有效"，并放在**最后**：

```go
// 推荐：用 error 表示
func Lookup(key string) (value string, err error)

// 推荐：当不需要额外说明时，用 bool 即可
// Lookup returns the value for key or ok=false if there is no mapping for key.
func Lookup(key string) (value string, ok bool)
```

## 六、先处理错误，主流程不嵌套

### 1. 错误处理放在前面，正常流程不放 `else`

> 来源：[Google § Indent error flow](https://google.github.io/styleguide/go/decisions.html#indent-error-flow)、[Effective Go § If](https://go.dev/doc/effective_go#if)。

错误一旦发生应**立即处理并跳出**（`return` / `continue` / `panic` / `log.Fatal`），后续主流程代码自然落在外层，不再缩进。这样读者顺着代码可读到的一条"快乐路径（happy path）"。

```go
// 推荐：错误先返回，主流程不嵌套
if err != nil {
    // error handling
    return
}
// normal code

// 不推荐：主流程被压进 else，可读性差
if err != nil {
    // error handling
} else {
    // normal code that looks abnormal due to indentation
}
```

同样的写法也适用于条件判定 + 早退场景（`return` / `continue` / `log.Fatal`），尽量让最常走的"正常路径"留在最外层。

### 2. 跨多行使用的变量不要写在 `if` 初始化里

如果某个变量后续要在多行代码中使用，**不要**塞进 `if x, err := f(); err != nil { ... } else { ... }` 的初始化里——这会把 `x` 的作用域限制在 `if` 块内，迫使主流程整体缩进。

```go
// 推荐：x 的作用域是整个函数，主流程不缩进
x, err := f()
if err != nil {
    // error handling
    return
}
// lots of code that uses x
// across multiple lines

// 不推荐：x 只在 else 里可见，正常流程被迫缩进
if x, err := f(); err != nil {
    // error handling
    return
} else {
    // lots of code that uses x
    // across multiple lines
}
```

> 反过来，如果变量**只在 `if` 块内使用**，`if x, err := f(); err != nil { ... }` 的写法依然推荐，能把作用域收敛得更小。

## 七、错误命名

### 1. 全局错误变量：导出用 `Err` 前缀，非导出用 `err` 前缀

对存储为**全局变量**的错误值（用 `errors.New` / `fmt.Errorf` 创建、供调用方用 `errors.Is` 匹配的 sentinel），按是否导出选择前缀：

- **导出**：`Err` 前缀
- **非导出**：`err` 前缀

```go
var (
    // 导出：调用方可用 errors.Is(err, foo.ErrBrokenLink) 匹配
    ErrBrokenLink   = errors.New("link is broken")
    ErrCouldNotOpen = errors.New("could not open")

    // 非导出：只在包内用 errors.Is(err, errNotFound) 匹配
    errNotFound = errors.New("not found")
)
```

> 一旦导出 `ErrXxx`，它就成为包公开 API 的一部分，**改名 / 删除属于破坏性变更**。能不导出就不导出。

### 2. 自定义错误类型：使用 `Error` 后缀

当错误信息包含**动态字段**、且调用方需要用 `errors.As` / `errors.AsType` 提取这些字段时，应定义**自定义错误类型**，类型名使用 `Error` 后缀：

- **导出**：`PascalCase` + `Error` 后缀
- **非导出**：`camelCase` + `Error` 后缀

```go
// 导出：调用方可用 errors.As / errors.AsType 匹配
type NotFoundError struct {
    File string
}

func (e *NotFoundError) Error() string {
    return fmt.Sprintf("file %q not found", e.File)
}

// 非导出：只在包内使用
type resolveError struct {
    Path string
}

func (e *resolveError) Error() string {
    return fmt.Sprintf("resolve %q", e.Path)
}
```

## 八、哨兵错误及其替代方案

> 来源：[Uber § Errors](https://github.com/uber-go/guide/blob/master/style.md#errors)、Dave Cheney [Don't just check errors, handle them gracefully](https://dave.cheney.net/2016/04/27/dont-just-check-errors-handle-them-gracefully)。

### 1. 什么是哨兵错误

**哨兵错误（sentinel error）** 是在包级别预先定义、可被调用方用 `errors.Is` 匹配的**固定错误变量**。命名按 [§7.1](#1-全局错误变量导出用-err-前缀非导出用-err-前缀) 走 `Err` / `err` 前缀。标准库随处可见：

```go
// io 包
var EOF = errors.New("EOF")

// sql 包
var ErrNoRows = errors.New("sql: no rows in result set")
```

调用方拿自己的错误和它对照，匹配上了就走对应分支：

```go
if errors.Is(err, sql.ErrNoRows) {
    // 走"没查到"的分支
}
```

### 2. 何时使用哨兵错误

仅当**全部**满足以下条件时再选择哨兵：

- 错误本身只是一个"无参状态信号"（如 EOF、连接已关闭、事务已提交）；
- 调用方只关心"是不是这个错误"，**不需要从中取任何字段**；
- 错误长期稳定，未来不会拆分或附加信息。

任一条件不满足，应优先考虑下面的替代方案。

### 3. 哨兵错误的代价

#### (a) 一旦导出就不能回收

`ErrXxx` 是包的公开 API，删除、改名、调整 wrap 行为均为破坏性变更。能不导出就不导出。

#### (b) 携带不了上下文

`errors.New("not found")` 只能告诉调用方"没找到"，无法告诉**没找到的是什么**。要带字段就只能换成自定义错误类型。

#### (c) 跨层判错让"包名"泄漏到上层

哨兵错误的"身份"绑定在**定义它的包**上。考虑典型调用链：

```text
handler ──► service ──► repository（定义 ErrNotFound）
```

如果 handler 直接写 `errors.Is(err, repository.ErrNotFound)`，那么 handler 就被迫 import repository——而它本来只应该认识 service。判错的人**通常不止直接调用方**，哨兵越多，依赖蔓延越严重，还容易在并列业务模块之间构成循环依赖。

> 直接调用方 import 被调用方是天然的；问题在于**跨层 / 跨模块判错时**，会把底层包名拉到本不该认识它的位置。

#### (d) 容易被滥用成全局错误码表

把所有错误都堆成 `var ErrXxx = errors.New(...)`，最后变成一张臃肿的错误码表，反而不如自定义错误类型清晰。

### 4. 替代方案一：自定义错误类型

调用方需要"识别错误 + 取字段"时，使用 [§7.2](#2-自定义错误类型使用-error-后缀) 的自定义错误类型，配合 `errors.As` / `errors.AsType` 提取字段。详见 [§9.5 动态错误：`fmt.Errorf` 与自定义错误类型的取舍](#5-动态错误fmterrorf-与自定义错误类型的取舍)。

### 5. 替代方案二：行为接口 + 判别函数（推崇）

跨层、跨模块判错时，**让上层依赖"错误能做什么"，而不是"错误是谁定义的"**。标准库 `net.Error`、`os.IsNotExist` / `os.IsPermission` 都是这个模式。

**三步走**：

(1) 错误类型多加一个语义化判别方法。

```go
// package repository
type NotFoundError struct {
    Entity string
    ID     string
}

func (e *NotFoundError) Error() string  { /* ... */ }
func (e *NotFoundError) NotFound() bool { return true }
```

(2) 在「使用错误」的一侧（调用方包或公共薄包）定义**未导出**的行为接口和**导出**的判别函数。

```go
// package errs
type notFoundError interface{ NotFound() bool }

func IsNotFound(err error) bool {
    var e notFoundError
    return errors.As(err, &e) && e.NotFound()
}
```

(3) 调用方只用判别函数判错，不依赖具体类型。

```go
if errs.IsNotFound(err) {
    return c.JSON(404, "not found")
}
```

**收益**：

- 调用方代码里**不再出现 repository 包名**，依赖只朝下，不反向泄漏；
- repository 想换实现、换包名、换错误类型，只要新错误仍实现 `NotFound()`，上层一行不用改；
- 接口未导出，判别细节不进入公开 API，未来可随时调整。

> **接口为什么要未导出？** 避免调用方绕过判别函数直接做类型断言，把内部约定变成公开 API；对外只承诺 `IsXxx` 判别函数本身。

### 6. 一个函数可能返回多种错误

`error` 只是**一个**返回值，但它可以承载多种"身份"。按场景挑工具：

| 真实意图 | 选择 |
| --- | --- |
| 一次调用只可能是其中一种 | 按优先级挨个 `if` 判别即可 |
| 一个错误"既是 A 又是 B" | 同一个错误类型**实现多个行为方法**（参考 `net.Error` 的 `Timeout()` + `Temporary()`） |
| 多个独立错误同时发生（批处理、并发） | `errors.Join(errs...)`（Go 1.20+） |
| 错误层层包装，每层语义不同 | 直接用 `%w` 包装，`errors.Is/As/AsType` 自动穿透 |

`errors.Join` 配合判别函数天然支持"部分失败"语义：

```go
func BatchCreate(users []User) error {
    var errs []error
    for _, u := range users {
        if err := repo.CreateOne(u); err != nil {
            errs = append(errs, fmt.Errorf("user %q: %w", u.Name, err))
        }
    }
    return errors.Join(errs...) // 全部成功时返回 nil
}
```

调用方依然按行为问：

```go
switch err := svc.BatchCreate(users); {
case err == nil:
    // ok
case errs.IsConflict(err) && errs.IsNotFound(err):
    // 部分冲突、部分找不到 → 207 Multi-Status
case errs.IsConflict(err):
    // 仅有冲突 → 409
case errs.IsNotFound(err):
    // 仅有未找到 → 404
default:
    // 其它 → 500
}
```

### 7. 选择一览

| 场景 | 推荐 |
| --- | --- |
| 无上下文、稳定、跨包共享的"状态信号" | **哨兵错误** |
| 错误需要携带字段供调用方使用 | **自定义错误类型** + `errors.As` / `errors.AsType` |
| 跨层 / 跨模块判错，希望调用方不依赖具体包 | **行为接口 + 判别函数** |
| 错误只是被传递、上层不判定 | 直接 `fmt.Errorf("...: %w", err)`，**不要新增任何哨兵或类型** |

## 九、错误包装与判定

### 1. 错误包装的三种选择：原样返回 / `%w` / `%v`

> 来源：[Uber § Error Wrapping](https://github.com/uber-go/guide/blob/master/style.md#error-wrapping) 与 [Google § Placement of %w in errors](https://google.github.io/styleguide/go/best-practices.html#placement-of-w-in-errors) 均有明文。

调用其它函数遇到错误时，有三种处理方式，按"是否需要补上下文、是否允许调用方追溯底层错误"选择：

| 写法 | 适用场景 | 备注 |
| --- | --- | --- |
| `return err` 原样返回 | 底层错误已经足够定位，且本层没有额外上下文可加 | 保留原始类型与消息，不会引入"噪音上下文" |
| `fmt.Errorf("...: %w", err)` | 需要补上下文，且**允许**调用方用 `errors.Is` / `errors.As` / `errors.AsType` 追溯 | **会成为函数契约的一部分**，导出函数最好在文档/测试里说明会包含哪些错误 |
| `fmt.Errorf("...: %v", err)` | 需要补上下文，但**不希望**调用方依赖底层错误类型 | 调用方无法匹配底层错误，未来如需开放可平滑切换为 `%w` |

简而言之：**没新信息就原样抛；要加上下文就在 `%w` / `%v` 间二选一**。

### 2. `%w` 的位置

**包装上游普通错误**时（默认情况），按"自然语序"把 `%w` 放在**末尾**，让错误链从新到旧依次展开：

```go
// 推荐：上下文在前，%w 在尾
return fmt.Errorf("parse config %q: %w", path, err)
```

层层包装后的可读链如下：

```go
err1 := fmt.Errorf("err1")
err2 := fmt.Errorf("err2: %w", err1)
err3 := fmt.Errorf("err3: %w", err2)
fmt.Println(err3) // err3: err2: err1
// 由新到旧（newest-to-oldest）的错误链，自然反映层级
```

**包装哨兵错误**时（哨兵用作错误"类别"信号），把 `%w` 放在错误字符串**最前面**，让分类信息第一时间可见——这是上面规则的唯一例外：

```go
package parser

var ErrParse = errors.New("parse error")

// 推荐：分类哨兵放最前，第一眼看到错误类别
var ErrParseInvalidHeader = fmt.Errorf("%w: invalid header", ErrParse)

func parseHeader() error {
    err := checkHeader()
    return fmt.Errorf("%w: invalid character in header: %v", ErrParseInvalidHeader, err)
}

// 不推荐：分类信号被埋在末尾，阅读时要扫到结尾才知道错误类别
return fmt.Errorf("invalid character in header: %v: %w", err, ErrParse)
```

> 经验法则：**默认 `%w` 放尾部**；如果被包装的是用来分类的哨兵，则放头部。

### 3. `errors.Is` / `errors.As` / `errors.AsType` 判定

使用 `%w` 包装错误后，调用方可用 `errors.Is` / `errors.As` / `errors.AsType` 判定具体错误类型：

```go
data, err := readFile("config.yaml")
if err != nil {
    if errors.Is(err, os.ErrNotExist) {
        // 文件不存在，走默认配置
        return defaultConfig(), nil
    }
    return nil, err
}
```

要从错误链里提取自定义类型的字段，则用 `errors.As`：

```go
var pathErr *os.PathError
if errors.As(err, &pathErr) {
    log.Printf("path=%s op=%s", pathErr.Path, pathErr.Op)
}
```

**Go 1.26+：优先使用泛型版 `errors.AsType`。** Go 1.26 在 `errors` 包新增了 `As` 的泛型版本：

```go
func AsType[E error](err error) (E, bool)
```

它把"要匹配的类型"放到类型参数里，目标错误通过返回值给出，省去了"预声明变量 + 传指针"那一套样板：

```go
// Go 1.13+：传统写法
var pathErr *os.PathError
if errors.As(err, &pathErr) {
    log.Printf("path=%s op=%s", pathErr.Path, pathErr.Op)
}

// Go 1.26+：泛型版，类型显式、变量作用域收敛在 if 块内
if pathErr, ok := errors.AsType[*os.PathError](err); ok {
    log.Printf("path=%s op=%s", pathErr.Path, pathErr.Op)
}
```

相比 `errors.As`，`AsType` 有几个优势：

- **类型安全**：要匹配的类型在编译期确定，不会出现"传了 `*T` 还是 `**T`"这种 runtime panic。
- **更简洁**：变量在 `if` 内随 `ok` 一起声明，作用域自然收敛，调用方少写一行。
- **更快**：实现里不再依赖反射，按类型断言 + `Unwrap()` 遍历错误树。
- **语义完全等价**：同样支持深度优先遍历多 wrap 错误（`Unwrap() []error`），同样尊重自定义错误的 `As(any) bool` 方法。

> 官方在 `errors.As` 的文档里已经写明 "For most uses, prefer [AsType]"。`errors.As` 不会被废弃（仍用于 `target` 不实现 `error` 接口的少数场景），但**新代码默认优先 `AsType`**。
>
> 注意：`errors.Is`（按 sentinel 值比较）没有泛型版本，仍然继续用。

### 4. 不要用字符串匹配判定错误

错误字符串可能随时被改写，靠 `strings.Contains` / 正则匹配判错既脆弱又违反 API 契约。应通过 sentinel 变量或自定义类型判定。

```go
// 不推荐：脆弱的字符串匹配
if strings.Contains(err.Error(), "not found") { /* ... */ }

// 推荐：通过 sentinel
if errors.Is(err, ErrNotFound) { /* ... */ }
```

### 5. 动态错误：`fmt.Errorf` 与自定义错误类型的取舍

对于带动态参数（文件名、ID 等）的错误：

- 调用方**不需要**程序化匹配 → 用 `fmt.Errorf` 即可；
- 调用方**需要**匹配并提取字段 → 定义自定义错误类型（`XxxError`），让调用方通过 `errors.As` 取字段。

> **注意**：一旦导出错误变量（`ErrXxx`）或错误类型（`XxxError`），它们就成为包公共 API 的一部分，后续修改会破坏调用方。能不导出就不导出。

**调用方不需要匹配**：

```go
// package foo
func Open(file string) error {
    return fmt.Errorf("file %q not found", file)
}

// package bar
if err := foo.Open("testfile.txt"); err != nil {
    // 无法对错误做精细化处理
    panic("unknown error")
}
```

**调用方需要匹配并使用字段**：

```go
// package foo
type NotFoundError struct {
    File string
}

func (e *NotFoundError) Error() string {
    return fmt.Sprintf("file %q not found", e.File)
}

func Open(file string) error {
    return &NotFoundError{File: file}
}

// package bar
if err := foo.Open("testfile.txt"); err != nil {
    var notFound *NotFoundError
    if errors.As(err, &notFound) {
        // 拿到 notFound.File 后做针对性处理
    } else {
        panic("unknown error")
    }
}
```

## 十、错误的分层处理与记录

### 1. 分层职责：dao / service / handler

错误不应该让每一层都"啥都做一遍"，要为每一层划清边界：

| 层 | 职责 | 典型动作 |
| --- | --- | --- |
| **dao / 基础设施层** | 包装基础设施错误 | `return fmt.Errorf("query user: %w", err)` |
| **service / 领域层** | 包装为领域错误，必要时映射 sentinel | `return fmt.Errorf("load user profile: %w", err)` 或 `return ErrUserNotFound` |
| **handler / 控制层** | 统一记录日志、转换成 HTTP/RPC 响应或外部结果 | `log.Errorw(...)` + `http.Error(w, ..., 500)` |

简记：**底层抛出 → 中层包装 → 上层记录**。

### 2. 不要重复记录错误

每一层都对同一个错误 `log.Errorf(...)` 一遍，会带来：

- 日志被同一错误刷屏，掩盖真正有用的上下文；
- 排障时难以分辨「这是 N 个错误」还是「一个错误被打了 N 次」；
- 关键字段（如 traceID、参数）反而被淹没。

**原则**：一条错误链上**只在边界打一次日志**，其余层级只做包装与返回。

- **最外层 / 边界层**（HTTP handler、消息消费入口、定时任务入口）：决定记录日志、返回给客户端或转交监控。
- **中间层**：只做 `fmt.Errorf("...: %w", err)` 包装上下文，**不要打日志**。

```go
// 不推荐：每一层都打一遍日志
func repo() error {
    if err := db.Query(); err != nil {
        log.Errorf("db query failed: %v", err) // 第 1 次
        return err
    }
    return nil
}

func service() error {
    if err := repo(); err != nil {
        log.Errorf("repo failed: %v", err) // 第 2 次（重复）
        return err
    }
    return nil
}

func handler(w http.ResponseWriter, r *http.Request) {
    if err := service(); err != nil {
        log.Errorf("service failed: %v", err) // 第 3 次（重复）
        http.Error(w, "internal error", 500)
        return
    }
}

// 推荐：中间层只包装，边界层统一记录
func repo() error {
    if err := db.Query(); err != nil {
        return fmt.Errorf("query user: %w", err)
    }
    return nil
}

func service() error {
    if err := repo(); err != nil {
        return fmt.Errorf("load user profile: %w", err)
    }
    return nil
}

func handler(w http.ResponseWriter, r *http.Request) {
    if err := service(); err != nil {
        // 唯一一次记录，错误链通过 %w 已经完整保留
        log.Errorw("handle request failed",
            "trace_id", r.Header.Get("X-Trace-Id"),
            "err", err,
        )
        http.Error(w, "internal error", 500)
        return
    }
}
```

> 例外：跨进程/跨异步边界（如发往消息队列前、`defer recover` 中）可以再记录一次，因为错误链已无法靠 `%w` 透传。
>
> **关于 `log.Error` 的成本**：很多日志库的 `Error` 级别会立刻把缓冲刷盘（fsync），单条成本远高于 `Info` / `Warn`。决定用 Error 还是 Warn 时，原则是「错误级日志必须具备可操作性（actionable）」，而不是「比 Warn 更严重」就用 Error。被高频触发又无法处置的"错误"应该降级为 Warn，或用 `rate.Sometimes` 限频。

### 3. 一次处理：错误的几种正确处置方式

调用方拿到错误后，应当根据契约**只处理一次**，避免「先打日志又把 err 原样返回」这种重复处理。常见的几种正确姿势：

#### (a) 包装错误并返回（最常用，把"如何处理"交给更上层）

```go
u, err := getUser(id)
if err != nil {
    return fmt.Errorf("get user %q: %w", id, err)
}
```

#### (b) 记录错误并优雅降级（操作不是关键路径，失败不应中断主流程）

```go
if err := emitMetrics(); err != nil {
    // 打点失败不应该影响业务
    log.Printf("emit metrics: %v", err)
}
```

#### (c) 匹配特定错误并降级，其它情况照常返回

```go
tz, err := getUserTimeZone(id)
if err != nil {
    if errors.Is(err, ErrUserNotFound) {
        tz = time.UTC // 找不到用户时退化为 UTC
    } else {
        return fmt.Errorf("get user %q: %w", id, err)
    }
}
```

#### (d) 反例：记录后又把原 err 返回

```go
u, err := getUser(id)
if err != nil {
    // BAD：上层很可能也会记一遍，造成日志刷屏
    log.Printf("could not get user %q: %v", id, err)
    return err
}
```

## 十一、正反例对比

| 反例 | 原因 | 推荐做法 |
| --- | --- | --- |
| `data, _ := ioutil.ReadFile(f)` | 静默丢弃错误，调用方无法感知失败 | `data, err := ioutil.ReadFile(f)` 并显式处理 |
| `return errors.New("failed")` | 错误信息空泛，无法定位 | `return fmt.Errorf("parse config %q: %w", path, err)` |
| `fmt.Errorf("failed to create new store: %w", err)` | `failed to` 是废话前缀，包装时层层堆积 | `fmt.Errorf("new store: %w", err)` |
| `fmt.Errorf("could not open settings.txt: %v", err)`（底层 `err` 已含文件名） | 重复了底层错误已经提供的信息 | `fmt.Errorf("launch codes unavailable: %v", err)` |
| `errors.New("Something bad happened.")` | 错误字符串首字母大写 + 句末标点 | `errors.New("something bad happened")` |
| `func Open() *PathError` | 导出函数返回具体类型，触发 nil interface 陷阱 | `func Open() error`（文档里说明实际类型） |
| `func Lookup(key string) int`（用 `-1` 表示未找到） | 带内错误易被漏判，错误被归咎到下游函数 | `func Lookup(key string) (value string, ok bool)` |
| `if x, err := f(); err != nil { return } else { /* 多行使用 x */ }` | 主流程被压进 `else`，可读性差 | 先 `x, err := f()` 再 `if err != nil { return }`，主流程不缩进 |
| `var NotFound = errors.New("not found")` | 全局 sentinel 缺少 `Err` 前缀 | `var ErrNotFound = errors.New("not found")` |
| `type NotFound struct{ ... }`（实现了 `Error()`） | 自定义错误类型缺少 `Error` 后缀 | `type NotFoundError struct{ ... }` |
| `fmt.Errorf("invalid header: %w", ErrParse)`（`ErrParse` 是分类 sentinel） | 分类信号被埋在字符串后半段，可读性差 | `fmt.Errorf("%w: invalid header", ErrParse)` |
| `if strings.Contains(err.Error(), "not found") { ... }` | 通过字符串匹配区分错误，脆弱且违反契约 | `if errors.Is(err, ErrNotFound) { ... }` |
| `var p *PathError; errors.As(err, &p)`（Go 1.26+） | 写法啰嗦、易混淆传 `*T` 还是 `**T` | `p, ok := errors.AsType[*PathError](err)` |
| 每一层都 `log.Errorf(err)` 后再 `return err` | 同一错误被打多遍，刷屏掩盖上下文 | 中间层只 `%w` 包装；仅在边界层记录一次 |
| 调用方需要按 file 字段判错，被调方返回 `fmt.Errorf("file %q not found", f)` | 字符串无法被结构化匹配 | 定义 `NotFoundError{File}`，调用方用 `errors.As` 取字段 |
| 上层 handler 写 `errors.Is(err, repository.ErrNotFound)` | 哨兵让 repository 的包名泄漏到 handler，破坏分层 | 在 service 层映射成行为接口，handler 用 `errs.IsNotFound(err)` 判错 |
| 业务里几乎所有错误都定义成 `var ErrXxx = errors.New(...)` | 哨兵被滥用成全局错误码表，无法携带上下文 | 仅"无参状态信号"用哨兵；需要字段就用自定义错误类型 |
