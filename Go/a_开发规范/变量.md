# Go 变量声明与初始化规范

> 变量是程序最基础的构件，一个项目里变量的声明、初始化、作用域的写法是否统一，直接影响代码的可读性、健壮性与性能。本文围绕「**统一声明方式**」「**最小作用域**」「**避免隐式状态**」三个核心理念，整理 Go 变量声明与初始化的常见做法。

## 一、变量声明方式

Go 提供了 `var`、`:=`、`new()` 与复合字面量等多种声明方式，**选择哪一种取决于你是否需要赋初值、是否需要指针**。统一规则有助于让读者从声明形式上一眼看出意图。


| 场景            | 推荐写法                               |
| ------------- | ---------------------------------- |
| 局部变量 + 有非零初值  | `i := 42`                          |
| 任意位置 + 零值     | `var coords Point`                 |
| 已知所有初始元素的复合类型 | `primes := []int{2, 3, 5, 7}`      |
| 需要指向零值的指针     | `msg := new(pb.Bar)` 或 `&pb.Bar{}` |


### 1. 非零值优先使用 `:=`

为保持一致性，**将非零值赋给新变量时，优先使用 `:=` 而非 `var`**。

```go
// Good
i := 42

// Bad: 多余的类型/var 噪音
var i int = 42
```

### 2. 零值使用 `var`

零值声明用 `var` 表达"这个变量先声明、后赋值"的意图最清晰：

```go
// Good
var (
    coords Point
    magic  [4]byte
    primes []int
)
```

### 3. 指针：`new` 与 `&T{}`

当需要一个**指向零值的指针**时，`new(T)` 与 `&T{}` 等价，二者皆可。但 `new` 这个关键字本身可以提醒读者：**如果需要非零字段，就不能用空的复合字面量**。

```go
// Good
var (
    buf = new(bytes.Buffer) // 非空 Buffer 通常由构造函数初始化
    msg = new(pb.Message)   // 非空 proto 由 builder 或逐字段赋值初始化
)
```

### 4. 相关的声明放在一组

Go 支持把相似的声明放在一组，应优先采用分组形式：

```go
// Bad
const a = 1
const b = 2

var a = 1
var b = 2

type Area float64
type Volume float64
```

```go
// Good
const (
    a = 1
    b = 2
)

var (
    a = 1
    b = 2
)

type (
    Area   float64
    Volume float64
)
```

> `import` 的分组规则更细致（按官方 / 项目内部 / 第三方分组），详见 [import 分组规范](./import.md)。

### 5. 不要把不相关的声明放在一组

只把**相关**的声明放在一起：

```go
// Bad
type Operation int

const (
    Add Operation = iota + 1
    Subtract
    Multiply
    EnvVar = "MY_ENV"
)
```

```go
// Good
type Operation int

const (
    Add Operation = iota + 1
    Subtract
    Multiply
)

const EnvVar = "MY_ENV"
```

分组使用的位置没有限制，例如，可以在函数内部使用：

```go
// Bad
func f() string {
    red := color.New(0xff0000)
    green := color.New(0x00ff00)
    blue := color.New(0x0000ff)

    // ...
}
```

```go
// Good
func f() string {
    var (
        red   = color.New(0xff0000)
        green = color.New(0x00ff00)
        blue  = color.New(0x0000ff)
    )

    // ...
}
```

### 6. 函数内部的相邻变量声明

**例外**：函数内部相邻的变量声明应分组在一起，即使它们并不严格相关。

```go
// Bad
func (c *client) request() {
    caller := c.name
    format := "json"
    timeout := 5 * time.Second
    var err error
    // ...
}
```

```go
// Good
func (c *client) request() {
    var (
        caller  = c.name
        format  = "json"
        timeout = 5 * time.Second
        err     error
    )
    // ...
}
```

> 这条规则与上文 [§一-2 零值使用 `var`](#2-零值使用-var) 中"零值用 `var`、非零值用 `:=`"并不矛盾——只在多变量同时声明、可以读起来更整齐时使用 `var (...)` 块。

## 二、零值与显式初始化

Go 的零值设计让"声明即可用"成为常态，但**必须区分零值是否安全可用**。`slice`、`map`、`channel`、`func`、`interface`、`pointer` 的零值都是 `nil`，其中 **`map` 必须显式初始化才能写入**。

### 1. 零值声明的典型场景

场景一，作为反序列化的输出参数：

```go
// Good
var coords Point
if err := json.Unmarshal(data, &coords); err != nil {
    // ...
}
```

场景二，作为构造函数的输出指针：

```go
// Good
msg := new(pb.Bar) // 或 &pb.Bar{}
if err := proto.Unmarshal(data, msg); err != nil {
    // ...
}
```

### 2. map 必须显式初始化才能写入

> **重要提示**：各种 `map` 类型在被写入之前必须**显式初始化**；但从值为 `nil` 的 map 中读取数据是安全的（返回零值）。

```go
// Bad: 写入 nil map 会 panic
var m map[string]int
m["foo"] = 1 // panic: assignment to entry in nil map

// Good
m := map[string]int{}
// 或
m := make(map[string]int)
m["foo"] = 1
```

### 3. 函数返回指针时直接声明为指针

如果一个变量是由构造函数产出、或无论如何都必须通过地址访问，那么**最好从一开始就声明为指针类型**，避免来回取地址：

```go
// Good
func NewCounter(name string) *Counter {
    c := new(Counter) // &Counter{} 也可以
    registerCounter(name, c)
    return c
}
```

### 4. slice 零值即可使用（无需 `make`）

与 §二-2 的 map 形成对照：**`nil` 是一个长度为 0 的有效切片**，可以直接读、可以直接 `append`，因此无需用 `[]T{}` 或 `make` 提前显式初始化。

- **不要显式返回长度为 0 的切片**，应直接返回 `nil`：

  ```go
  // Bad
  if x == "" {
      return []int{}
  }
  ```

  ```go
  // Good
  if x == "" {
      return nil
  }
  ```

- **判断切片是否为空请用 `len(s) == 0`**，而不是与 `nil` 比较：

  ```go
  // Bad
  func isEmpty(s []string) bool {
      return s == nil
  }
  ```

  ```go
  // Good
  func isEmpty(s []string) bool {
      return len(s) == 0
  }
  ```

- **零值切片（`var` 声明）开箱即用，无需 `make`**：

  ```go
  // Bad
  nums := []int{}
  if add1 {
      nums = append(nums, 1)
  }
  ```

  ```go
  // Good
  var nums []int
  if add1 {
      nums = append(nums, 1)
  }
  ```

> **注意**：`nil` 切片虽然是有效切片，但**不等于长度为 0 的切片**——一个为 `nil`，另一个不是。在 JSON 序列化等场景下，二者表现可能不同。

## 三、复合字面量

### 1. 已知初始元素时使用复合字面量

**当你已经知道各个初始元素或字段的值时**，应使用复合字面量声明：

```go
// Good
var (
    coords   = Point{X: x, Y: y}
    magic    = [4]byte{'I', 'W', 'A', 'D'}
    primes   = []int{2, 3, 5, 7, 11}
    captains = map[string]string{"Kirk": "James Tiberius", "Picard": "Jean-Luc"}
)
```

### 2. 空值优先用零值或 `new`

相比之下，**用复合字面量表示空值或没有成员的值在视觉上比较杂乱**，不如直接使用零值或 `new`：

```go
// Bad: 视觉噪音多
var (
    primes = []int{}
    cache  = map[string]int{}
)

// Good: 用零值表达"空"
var (
    primes []int
    cache  map[string]int // 注意：map 在写入前仍需显式初始化
)
```

> **何时使用 `new`**：当需要一个指向零值的指针时，`new(T)` 与 `&T{}` 都可以；`new` 关键字本身就提示"这是零值，需要字段填充就别用空字面量"。

### 3. 使用字段名初始化结构体

初始化结构体时，**几乎总是应该指定字段名**，由 [go vet](https://golang.org/cmd/vet/) 强制检查。

```go
// Bad
k := User{"John", "Doe", true}
```

```go
// Good
k := User{
    FirstName: "John",
    LastName:  "Doe",
    Admin:     true,
}
```

> **例外**：表驱动测试中，当结构体字段不超过 3 个时，可省略字段名以让表格更紧凑。

```go
tests := []struct {
    op   Operation
    want string
}{
    {Add, "add"},
    {Subtract, "subtract"},
}
```

### 4. 省略零值字段

按字段名初始化时，**除非提供有意义的上下文，否则忽略零值字段**——让 Go 的零值机制自动填充。

```go
// Bad
user := User{
    FirstName:  "John",
    LastName:   "Doe",
    MiddleName: "",
    Admin:      false,
}
```

```go
// Good
user := User{
    FirstName: "John",
    LastName:  "Doe",
}
```

只在字段名能提供**有意义的语义上下文**时显式写出零值——例如表驱动测试，让每行都按相同列对齐：

```go
tests := []struct {
    give string
    want int
}{
    {give: "0", want: 0},
    // ...
}
```

## 四、循环中的变量声明

> 一个常见疑问：**在 `for` 循环里，每次迭代都新建变量好，还是循环外声明一次复用？**
>
> **结论：默认在循环内部声明**，除非有明确的性能或语义理由要外部声明。

### 1. 默认在循环内部声明

循环内声明有三个直接收益：

**(1) 作用域最小化**：变量只在需要的地方可见，避免泄漏到外层、被后续代码误用。

```go
// Good
for i := 0; i < n; i++ {
    x := compute(i)
    use(x)
}
```

```go
// Bad: x 泄漏到循环外，仍可被误用
var x int
for i := 0; i < n; i++ {
    x = compute(i)
    use(x)
}
// 这里 x 仍然可见
```

**(2) 避免"状态残留"bug**：循环外声明的复合类型（slice/map/struct）每次迭代都带着上次的数据，忘记重置就出 bug：

```go
// Bad: 累积了上一次的数据
var buf []int
for _, item := range items {
    buf = append(buf, item.values...)
    process(buf) // buf 在多次迭代之间不断变长
}

// Good: 每次循环都是干净的零值
for _, item := range items {
    var buf []int
    buf = append(buf, item.values...)
    process(buf)
}
```

**(3) 性能上几乎没差异**：Go 编译器有逃逸分析，没有逃逸的变量分配在**栈**上，循环内/外声明对栈帧布局影响极小。

### 2. 何时应在循环外声明

只有满足下列任意一条才值得这么做：

**(1) 复用大对象 / 减少堆分配**：变量逃逸到堆，且分配开销大（大 buffer、大 slice、大 map），可以外部声明 + 每次重置复用：

```go
// Good: 一次分配，循环复用底层数组
buf := make([]byte, 0, 4096)
for _, item := range items {
    buf = buf[:0] // 重置长度，保留容量
    buf = appendData(buf, item)
    write(buf)
}
```

更进一步的场景可考虑 `sync.Pool`。

**(2) 需要跨迭代维护状态**：累加器、上一次的值等本来就是循环间共享的语义：

```go
// Good: sum/prev 的语义就是跨迭代
sum := 0
prev := 0
for _, v := range nums {
    sum += v
    diff := v - prev
    _ = diff
    prev = v
}
```

这是**语义需要**，而不是"为了优化"。

### 3. Go 1.22 循环变量改动

这是很多人混淆的地方，需要区分清楚：


| 变量类型                                                          | Go 1.22 之前                       | Go 1.22 之后      |
| ------------------------------------------------------------- | -------------------------------- | --------------- |
| `for i := 0; i < n; i++` 的 `i`，`for i, v := range s` 的 `i, v` | **整个循环共用一个**，闭包 / goroutine 捕获有坑 | **每次迭代新建**，自动安全 |
| 循环体内 `x := ...` 自己声明的变量                                       | **一直都是每次迭代新建**                   | 不变              |


也就是说：**你在循环体内 `:=` 声明的变量，从来都是每次新建的**，不需要担心闭包问题。

Go 1.22 之前的经典 bug：

```go
// Bad (Go < 1.22): 所有 goroutine 看到的都是同一个 i, v
for i, v := range items {
    go func() {
        fmt.Println(i, v)
    }()
}

// 修复方式（Go < 1.22 必须显式拷贝一份）
for i, v := range items {
    i, v := i, v // 在循环体内声明新的同名变量
    go func() {
        fmt.Println(i, v)
    }()
}
```

Go 1.22 之后上面第一段代码已经是正确的，但**为了向后兼容**，写库代码时仍建议保留显式拷贝。

## 五、变量作用域

**核心原则：让变量的作用域尽可能小**。一个变量声明位置距离它被使用的位置越近，读者越容易理解它的生命周期。


| 场景           | 推荐写法                                     |
| ------------ | ---------------------------------------- |
| 仅在 `if` 内使用  | `if x, err := foo(); err == nil { ... }` |
| 仅在 `for` 内使用 | `for i := 0; i < n; i++ { ... }`（详见 §四）  |
| 跨多个分支使用      | 在最近的公共父作用域声明                             |
| 包级共享         | 放在 `var ( ... )` 块顶部，并在文档中说明用途           |


**优先使用带初始化语句的 `if` / `switch`**：

```go
// Good
if data, err := load(path); err != nil {
    return err
} else {
    process(data)
}

// Bad: data 和 err 泄漏到外层
data, err := load(path)
if err != nil {
    return err
}
process(data)
```

> 包级变量、`sync.Once`、`init` 等更广作用域的话题不在本文范围。
