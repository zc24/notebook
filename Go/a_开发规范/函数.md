## 一、简洁优先

- **用最少、清晰的代码实现功能**：一个函数只做一件事。
- **避免过度设计**：不引入不必要的抽象、依赖或复杂设计模式，优先使用标准库。
- **控制函数长度**：函数过长（例如超过 50 行）时，考虑拆分为多个更小的函数。
- **控制参数数量**：参数超过 3~4 个时考虑封装成结构体或使用 Option 模式（详见 [§3 参数控制](#三参数控制)）。

## 二、单一职责与函数长度

```go
// 不推荐：函数冗长，职责不清
func processDataAndSend(data []byte) error {
    // ... 很长很复杂的解析逻辑
    // ... 很长很复杂的处理逻辑
    // ... 很长很复杂的发送逻辑
    return nil
}

// 推荐：拆分为多个单一职责的函数
func processData(data []byte) (*ProcessedResult, error) {
    // 只负责解析
    return &ProcessedResult{}, nil
}

func sendResult(result *ProcessedResult) error {
    // 只负责发送
    return nil
}

// 主函数变得非常简洁
func mainLogic(data []byte) error {
    result, err := processData(data)
    if err != nil {
        return err
    }
    return sendResult(result)
}
```

## 三、参数控制

参数过多会带来几个问题：

- 调用方需要记忆参数顺序，容易传错位置；
- 新增字段必须改动所有调用点，难以演进；
- 难以从签名上区分必填项与可选项。

经验法则：**参数数量控制在 3~4 个以内**，超过则考虑下面两种方式。

### 1. 参数对象（Parameter Object）

把语义上属于同一组的参数打包成结构体，并用具名字段构造，避免位置依赖。

```go
// 不推荐：参数过多，调用点像一串魔法值
func CreateUser(name, email, phone, address, city, country string, age int, vip bool) error {
    // ...
    return nil
}

CreateUser("Tom", "tom@x.com", "13800000000", "No.1", "SH", "CN", 18, true)

// 推荐：封装为参数对象，调用点自解释
type CreateUserParams struct {
    Name    string
    Email   string
    Phone   string
    Address string
    City    string
    Country string
    Age     int
    VIP     bool
}

func CreateUser(p CreateUserParams) error {
    // ...
    return nil
}

CreateUser(CreateUserParams{
    Name:    "Tom",
    Email:   "tom@x.com",
    Phone:   "13800000000",
    Address: "No.1",
    City:    "SH",
    Country: "CN",
    Age:     18,
    VIP:     true,
})
```

### 2. Functional Options

适用于「少量必填 + 大量可选」的构造场景，例如 HTTP Client、Server 等。

```go
type Server struct {
    addr    string
    timeout time.Duration
    tls     bool
}

type Option func(*Server)

func WithTimeout(d time.Duration) Option { return func(s *Server) { s.timeout = d } }
func WithTLS() Option                    { return func(s *Server) { s.tls = true } }

func NewServer(addr string, opts ...Option) *Server {
    s := &Server{addr: addr, timeout: 5 * time.Second} // 合理默认值
    for _, opt := range opts {
        opt(s)
    }
    return s
}

// 调用方按需启用，签名稳定，扩展不破坏旧调用
srv := NewServer(":8080", WithTimeout(10*time.Second), WithTLS())
```

> 经验：必填项放普通参数，可选项放 `Option`；不要把所有字段都做成 `Option`，否则反而失去类型约束。

### 3. 传值优于传指针

**不要仅仅为了节省几个字节而把参数声明成指针。** 如果函数在整个调用过程中始终只通过 `*x` 来引用其参数 `x`，那么这个参数就不应该是指针。

常见反例：把字符串（`*string`）、接口值（`*io.Reader`）等本身就是固定大小的值传成指针——值本身可以直接传递。

```go
// 不推荐：本身就是固定大小的值，没必要传指针
func handle(name *string, r *io.Reader) { /* ... */ }

// 推荐：直接传值，调用方更易用、零额外解引用
func handle(name string, r io.Reader) { /* ... */ }
```

> 例外：这条建议**不适用于大型结构体**，也不适用于将来可能增长的小型结构体——这类场景仍应通过指针传递以避免拷贝开销。

## 四、返回值设计

### 1. 命名结果参数（Named Result Parameters）

命名结果参数最直接的副作用是会出现在 `godoc` 里。如果命名只是把类型名再重复一遍，就只会让签名更啰嗦：

```go
// 不推荐：node、err 只是把类型再写一遍，godoc 里冗余
func (n *Node) Parent1() (node *Node)              { /* ... */ }
func (n *Node) Parent2() (node *Node, err error)   { /* ... */ }

// 推荐：直接给类型即可
func (n *Node) Parent1() *Node           { /* ... */ }
func (n *Node) Parent2() (*Node, error)  { /* ... */ }
```

但**以下情况，命名结果参数是有价值的**：

- 函数返回**两个或更多同类型的值**，名字可以帮助区分它们的语义；
- 返回值的含义**从上下文无法直接看出**；
- 需要在 `defer` 闭包里**修改返回值**；
- 调用方需要根据特定的结果参数来采取行动（例如 `context` 返回的 `cancel` 函数），命名有助于明确应采取何种行动。

```go
// 推荐：通过命名把两个 float64 的语义讲清楚
// Location returns f's latitude and longitude.
// Negative values mean south and west, respectively.
func (f *Foo) Location() (lat, long float64, err error) { /* ... */ }
```

> 不要仅仅为了「在函数内部省一行 `var`」就给返回值命名——这是在用 API 的清晰度换实现的一点点便利，不划算。

提示：在函数签名中，类型描述往往比函数名称更清晰明了。

### 2. 裸返回（Naked Returns）

不带参数的 `return` 语句会返回命名的返回值，被称为「裸返回」。

```go
func split(sum int) (x, y int) {
    x = sum * 4 / 9
    y = sum - x
    return // 等价于 return x, y
}
```

使用准则：

- **函数只有几行**时，裸返回可以接受；
- 函数一旦达到中等规模，**请显式 `return x, y`**——否则读者要往上翻才能知道返回了什么；
- **不要仅仅为了能写裸返回就给返回值命名**：文档与可读性永远优先于函数体里少写一两行；
- 如果某个结果参数的值需要在延迟执行的代码块中被修改，那么给该参数命名是完全可以接受的。

## 五、作用域与代码路径

### 1. 最小作用域

- **最小作用域**：变量应在最近使用的地方声明，并限制其作用域。
- **优先使用短变量声明 `:=`**：让变量尽量贴近第一次使用的位置。
- **避免提前声明**：函数顶部一次性声明大量变量，会扩大其生命周期与可见范围。

```go
// 不推荐：变量过早声明，作用域过大
func handle(items []Item) error {
    var total int
    var err error
    for _, it := range items {
        total += it.Value
    }
    if total > 100 {
        err = doSomething(total)
    }
    return err
}

// 推荐：变量贴近使用点，作用域最小
func handle(items []Item) error {
    total := 0
    for _, it := range items {
        total += it.Value
    }
    if total > 100 {
        if err := doSomething(total); err != nil {
            return err
        }
    }
    return nil
}
```

### 2. 错误处理优先，正常路径不缩进

尽量保持正常代码路径的缩进最小，把错误处理放进 `if` 块中并尽早 `return`。这样可以让读者一眼扫到正常路径，提升可读性。

```go
// 不推荐：正常路径被 else 包住，缩进越来越深
if err != nil {
    // error handling
} else {
    // normal code
}

// 推荐：错误处理先返回，正常路径平铺
if err != nil {
    // error handling
    return // 或 continue 等
}
// normal code
```

### 3. `if` 初始化语句的写法

如果 `if` 的初始化语句声明了一个变量，而该变量需要在 `if` 块之外继续使用，请把短变量声明拆到单独一行——这样既能尽早 `return` 处理错误，又不会把变量困在 `if` 作用域内。

```go
// 不推荐：x 被困在 else 里，正常路径反而被嵌套
if x, err := f(); err != nil {
    // error handling
    return
} else {
    // use x
}

// 推荐：声明独立成行，错误处理先 return，x 在外部正常使用
x, err := f()
if err != nil {
    // error handling
    return
}
// use x
```

> 「实际值 op 预期值」的条件顺序约定（变量在左、常量在右）属于条件语句格式问题，统一在 [§7.6](#6-条件判断顺序实际值在左预期值在右避免尤达式条件句) 介绍。

## 六、函数格式化

函数参数列表常常会让 Go 源代码中的某些行变得极其冗长。由于参数列表位于缩进变化的位置之前，简单按列宽换行会让后续行看起来像函数体的一部分，造成阅读上的混淆。

```go
// Bad: 参数换行后与函数体缩进对齐，难以区分
func (r *SomeType) SomeLongFunctionName(foo1, foo2, foo3 string,
    foo4, foo5, foo6 int) {
    foo7 := bar(foo1)
    // ...
}
```

### 1. 提取局部变量简化长签名

通过提取局部变量，通常可以简化代码中的表达式。

```go
// Good:
local := helper(some, parameters, here)
good := foo.Call(list, of, parameters, local)
```

### 2. 不要仅按行数决定是否换行

函数和方法的调用不应仅仅根据行数来决定是否分开。

```go
// Good:
good := foo.Call(long, list, of, parameters, all, on, one, line)

// Bad:
bad := foo.Call(long, list, of, parameters,
    with, arbitrary, line, breaks)
```

### 3. 避免在参数上加内联注释

尽可能避免在特定函数参数上添加内联注释。可以使用选项结构体或在函数文档中提供更详细的说明。

```go
// Good:
good := server.New(ctx, server.Options{Port: 42})

// Bad:
bad := server.New(
    ctx,
    42, // Port
)
```

### 4. 按语义而非行长换行

如果无法更改 API 的格式，或者本地调用方式比较特殊，那么只要有助于理解调用逻辑，完全可以添加换行符让代码更易读。

```go
// Good: 按顶点坐标和颜色分组，而非按列边界
canvas.RenderHeptagon(fillColor,
    x0, y0, vertexColor0,
    x1, y1, vertexColor1,
    x2, y2, vertexColor2,
    x3, y3, vertexColor3,
    x4, y4, vertexColor4,
    x5, y5, vertexColor5,
    x6, y6, vertexColor6,
)
```

### 5. 长格式字符串的换行

为了符合行长度要求而拆分过长的字符串是不合适的。对于包含此类字符串的函数，可以在字符串格式之后换行，然后将各个参数放在下一行或后续行中。决定在哪里换行时，应依据输入数据的语义结构，而不仅仅是行长度。

```go
// Good:
log.Warningf("Database key (%q, %d, %q) incompatible in transaction started by (%q, %d, %q)",
    currentCustomer, currentOffset, currentKey,
    txCustomer, txOffset, txKey)

// Bad:
log.Warningf("Database key (%q, %d, %q) incompatible in"+
    " transaction started by (%q, %d, %q)",
    currentCustomer, currentOffset, currentKey, txCustomer,
    txOffset, txKey)
```

### 6. 原始字符串字面值，避免转义

Go 支持 [原始字符串字面值](https://golang.org/ref/spec#raw_string_lit)（用反引号 `` ` `` 包围），可以跨多行、不需要转义引号或反斜杠。在需要大量转义时优先使用：

```go
// Bad
wantError := "unknown name:\"test\""
```

```go
// Good
wantError := `unknown name:"test"`
```

## 七、条件语句与循环结构

### 1. `if` 语句不应换行

多行的 `if` 子句可能会导致缩进混乱。

```go
// Bad:
// 第二个 if 子句与 if 块内代码对齐，造成缩进混淆
if db.CurrentStatusIs(db.InTransaction) &&
    db.ValuesEqual(db.TransactionKey(), row.Key()) {
    return db.Errorf(db.TransactionError, "query failed: row (%v): key does not match transaction key", row)
}
```

如果不需要短路求值，可以直接提取布尔表达式的值：

```go
// Good:
inTransaction := db.CurrentStatusIs(db.InTransaction)
keysMatch := db.ValuesEqual(db.TransactionKey(), row.Key())
if inTransaction && keysMatch {
    return db.Error(db.TransactionError, "query failed: row (%v): key does not match transaction key", row)
}
```

### 2. 提取重复子表达式

当条件中重复出现相同子表达式时，可以提取本地变量：

```go
// Good:
uid := user.GetUniqueUserID()
if db.UserIsAdmin(uid) || db.UserHasPermission(uid, perms.ViewServerConfig) || db.UserHasPermission(uid, perms.CreateGroup) {
    // ...
}

// Bad:
if db.UserIsAdmin(user.GetUniqueUserID()) || db.UserHasPermission(user.GetUniqueUserID(), perms.ViewServerConfig) || db.UserHasPermission(user.GetUniqueUserID(), perms.CreateGroup) {
    // ...
}
```

### 3. 包含闭包或多行字面量的 `if`

`if` 中包含闭包或多行结构字面量时，必须确保大括号配对正确，避免缩进混淆。

```go
// Good:
if err := db.RunInTransaction(func(tx *db.TX) error {
    return tx.Execute(userUpdate, x, y, z)
}); err != nil {
    return fmt.Errorf("user update failed: %s", err)
}

// Good:
if _, err := client.Update(ctx, &upb.UserUpdateRequest{
    ID:   userID,
    User: user,
}); err != nil {
    return fmt.Errorf("user update failed: %s", err)
}
```

### 4. 不要在 `for` 语句中人为换行

不要试图在 `for` 语句中人为地添加换行。如果无法以更优雅的方式重构，那就让那一行保持原样。

```go
// Good:
for i, max := 0, collection.Size(); i < max && !collection.HasPendingWriters(); i++ {
    // ...
}
```

不过，通常情况下还是会有这种重构方式：

```go
// Good:
for i, max := 0, collection.Size(); i < max; i++ {
    if collection.HasPendingWriters() {
        break
    }
    // ...
}
```

### 5. `switch` / `case` 的换行

`switch` 和 `case` 语句应保持在同一行上。如果该行太长，建议将所有内容缩进，并用空行将它们分隔，避免缩进混乱：

```go
// Good:
switch db.TransactionStatus() {
case
    db.TransactionStarting,
    db.TransactionActive,
    db.TransactionWaiting,
    db.TransactionCommitted:

    // ...
case db.NoTransaction:
    // ...
default:
    // ...
}
```

### 6. 条件判断顺序：实际值在左、预期值在右（避免「尤达式条件句」）

Go 中条件判断推荐写成 **「实际值 op 预期值」**——被检查的变量放在左侧、常量或预期值放在右侧。这种写法贴近自然语言阅读顺序（「如果 status code 不等于 200」），也便于按变量名搜索；反之把常量放在前面，被称为 **「尤达式条件句」**（Yoda condition），表述不清，应当避免。

```go
// Good: 变量在左，常量在右
if resp.StatusCode != 200 {
    // ...
}

if result == "foo" {
    // ...
}

// Bad: 把常量放前面（尤达式条件句）
if 200 != resp.StatusCode {
    // ...
}

if "foo" == result {
    // ...
}
```

### 7. 不必要的 `else`

如果在 `if` 的两个分支中都给同一个变量赋值，可以将其简化为一个 `if`，把"默认值"放在声明里：

```go
// Bad
var a int
if b {
    a = 100
} else {
    a = 10
}
```

```go
// Good
a := 10
if b {
    a = 100
}
```

> 与之相关的"错误处理优先、正常路径不缩进"的规则，详见 [§五-2 错误处理优先](#2-错误处理优先正常路径不缩进)。

## 八、函数与方法命名

在为函数或方法命名时，请先想清楚该名称会出现在何种上下文中。**好的命名应当贴合调用点的语境**，而不是把签名里已有的信息再写一遍。

### 1. 避免不必要的重复

在函数或方法名中，**通常可以省略以下内容**（在不会引起歧义、与同名函数无碰撞的前提下）：

- 输入和输出的类型；
- 方法的接收者类型；
- 输入或输出是否为指针。

**① 不要重复包名。** 调用方写的是 `yamlconfig.Parse`，名字里再带 `YAML` 就是冗余。

```go
// Bad:
package yamlconfig

func ParseYAMLConfig(input string) (*Config, error)

// Good:
package yamlconfig

func Parse(input string) (*Config, error)
```

**② 不要重复接收者类型名。** 调用方写的是 `c.WriteTo(w)`，名字里再带 `Config` 同样冗余。

```go
// Bad:
func (c *Config) WriteConfigTo(w io.Writer) (int64, error)

// Good:
func (c *Config) WriteTo(w io.Writer) (int64, error)
```

**③ 不要重复参数名。** 参数顺序已经表达了语义，名字不必重复。

```go
// Bad:
func OverrideFirstWithSecond(dest, source *Config) error

// Good:
func Override(dest, source *Config) error
```

**④ 不要重复返回值类型。** 返回值类型在签名里已经写明，函数名里没必要再写一遍。

```go
// Bad:
func TransformToJSON(input *Config) *jsonconfig.Config

// Good:
func Transform(input *Config) *jsonconfig.Config
```

**⑤ 例外：当需要区分相似函数时，可以保留必要的修饰词。**

```go
// Good: 通过 Text / Binary 区分两个相似函数
func (c *Config) WriteTextTo(w io.Writer) (int64, error)
func (c *Config) WriteBinaryTo(w io.Writer) (int64, error)
```

### 2. 命名惯例

**① 返回值的函数用「名词」命名，避免 `Get` 前缀。**

```go
// Good:
func (c *Config) JobName(key string) (value string, ok bool)

// Bad:
func (c *Config) GetJobName(key string) (value string, ok bool)
```

**② 执行某种动作的函数用「动词」命名。**

```go
// Good:
func (c *Config) WriteDetail(w io.Writer) (int64, error)
```

**③ 仅因类型不同而区分的函数，名称中带上类型信息。**

```go
// Good:
func ParseInt(input string) (int, error)
func ParseInt64(input string) (int64, error)
func AppendInt(buf []byte, value int) []byte
func AppendInt64(buf []byte, value int64) []byte
```

**④ 如果存在明确的「主要」版本，主版本的名称可以省略类型信息。**

```go
// Good: Marshal 是主版本，输出 []byte；MarshalText 输出 string
func (c *Config) Marshal() ([]byte, error)
func (c *Config) MarshalText() (string, error)
```

## 九、Don't panic（不要使用 panic 处理常规错误）

- 不要使用 `panic` 进行常规错误处理。应使用 `error` 以及多个返回值来处理错误。
- 在 `package main` 和初始化代码中，对于会导致程序终止的错误（例如配置无效），可以考虑使用 `log.Exit`。在很多此类场景下，堆栈跟踪信息对用户并没有帮助。
  - 注意：`log.Exit` 会调用 `os.Exit`，所有被延迟执行的 `defer` 函数都不会运行。
- 对于「不可能发生」的错误（即在代码审查与测试阶段本应被发现的错误），函数可以返回 `error`，或调用 `log.Fatal`。

## 十、Must 函数

那些在程序出错时用于终止程序的辅助函数，其命名规则为 `MustXYZ`（或 `mustXYZ`）。这类函数通常应在程序启动早期被调用，而不应用于需要常规错误处理的场景（例如处理用户输入）。

最常见的使用场景：在包初始化时，用于初始化包级变量（例如 `template.Must` 与 `regexp.MustCompile`）。

```go
// Good:
func MustParse(version string) *Version {
    v, err := Parse(version)
    if err != nil {
        panic(fmt.Sprintf("MustParse(%q) = _, %v", version, err))
    }
    return v
}

// 包级「常量」。如果想用 Parse，则需要在 init 中赋值
var DefaultVersion = MustParse("1.2.3")
```

这种模式的优点：辅助函数可以在「值」上下文中被调用。

> 注意：`Must` 函数不应在难以确保错误能被捕获的场合使用，也不应在需要检查错误的上下文中使用（例如请求处理程序）。
>
> - 对于常量输入，`Must` 有助于确保参数格式正确；
> - 对于非常量输入，应优先使用常规错误处理机制，确保错误能被妥善处理或传递。

## 十一、flag 库

参数名采用蛇形命名法（snake_case），存储该 flag 值的变量名采用驼峰式命名法（camelCase）。

```go
// Good:
var (
    pollInterval = flag.Duration("poll_interval", time.Minute, "Interval to use for polling.")
)
```

## 十二、Context 库

- 当被传递给某个函数或方法时，`context.Context` **始终是第一个参数**。
- 在调用链中，代码很少需要使用 `context.Background()` 来创建自己的上下文。**除非所获取的上下文不正确，否则应始终优先使用调用方提供的 context**。
- **不要创建自定义的上下文类型**，也不要在函数签名中使用除 `context.Context` 之外的其他接口。**这一规则没有例外**。

## 十三、Printf 风格的格式字符串与函数命名

`go vet` 可以对 Printf 系列函数的格式字符串做静态检查（参数个数、类型、`%` 占位符匹配等），前提是格式字符串和函数名都遵循一定的约定。

### 1. 函数外的格式字符串声明为 `const`

如果在函数外声明 `Printf`-style 函数的格式字符串，请使用 `const`。这样 `go vet` 才能在编译期对格式字符串执行**静态分析**——`var` 声明的字符串运行时可能被改写，无法静态校验。

```go
// Bad
msg := "unexpected values %v, %v\n"
fmt.Printf(msg, 1, 2)
```

```go
// Good
const msg = "unexpected values %v, %v\n"
fmt.Printf(msg, 1, 2)
```

### 2. 自定义 Printf 函数以 `f` 结尾

声明自己的 `Printf`-style 函数时，要让 `go vet` 能识别并检查格式字符串。

- **优先使用预定义名称**（`Printf`、`Sprintf`、`Errorf`、`Fatalf` 等），`go vet` 默认会检查它们。详见 [`go vet` Printf 系列](https://golang.org/cmd/vet/#hdr-Printf_family)。
- **必须使用自定义名称时，以 `f` 结尾**，例如 `Wrapf` 而非 `Wrap`。`go vet` 可以通过 `-printfuncs` 检查特定的 Printf 样式名称，但**名称必须以 `f` 结尾**：

  ```bash
  go vet -printfuncs=wrapf,statusf
  ```

> 关于 Printf 函数的格式参数检查，参考 [go vet: Printf family check](https://kuzminva.wordpress.com/2017/11/07/go-vet-printf-family-check/)。
