## 一、简洁优先

- **用最少、清晰的代码实现功能**：一个函数只做一件事。
- **避免过度设计**：不引入不必要的抽象、依赖或复杂设计模式，优先使用标准库。
- **函数长度**：函数过长（例如超过 50 行）时，考虑拆分为多个更小的函数。
- **参数别失控**：参数超过 3~4 个时考虑封装成结构体或使用 Option 模式（详见 [参数控制](#三参数控制)）。

## 二、单一职责与函数长度

```go
// 不推荐：函数冗长，职责不清
func processDataAndSend(data []byte) error {
    // ... 很长很复杂的解析逻辑
    // ... 很长很复杂的处理逻辑
    // ... 很长很复杂的发送逻辑
    return nil
}

// 推荐：拆分为多个单一职责的函数
func processData(data []byte) (*ProcessedResult, error) {
    // 只负责解析
    return &ProcessedResult{}, nil
}

func sendResult(result *ProcessedResult) error {
    // 只负责发送
    return nil
}

// 主函数变得非常简洁
func mainLogic(data []byte) error {
    result, err := processData(data)
    if err != nil {
        return err
    }
    return sendResult(result)
}
```

## 三、参数控制

参数过多会带来几个问题：

- 调用方需要记忆参数顺序，容易传错位置；
- 新增字段必须改动所有调用点，难以演进；
- 难以从签名上区分必填项与可选项。

经验法则：**参数数量控制在 3~4 个以内**，超过则考虑下面两种方式。

### 1. 参数对象（Parameter Object）

把语义上属于同一组的参数打包成结构体，并用具名字段构造，避免位置依赖。

```go
// 不推荐：参数过多，调用点像一串魔法值
func CreateUser(name, email, phone, address, city, country string, age int, vip bool) error {
    // ...
    return nil
}

CreateUser("Tom", "tom@x.com", "13800000000", "No.1", "SH", "CN", 18, true)

// 推荐：封装为参数对象，调用点自解释
type CreateUserParams struct {
    Name    string
    Email   string
    Phone   string
    Address string
    City    string
    Country string
    Age     int
    VIP     bool
}

func CreateUser(p CreateUserParams) error {
    // ...
    return nil
}

CreateUser(CreateUserParams{
    Name:    "Tom",
    Email:   "tom@x.com",
    Phone:   "13800000000",
    Address: "No.1",
    City:    "SH",
    Country: "CN",
    Age:     18,
    VIP:     true,
})
```

### 2. Functional Options

适用于「少量必填 + 大量可选」的构造场景，例如 HTTP Client、Server 等。

```go
type Server struct {
    addr    string
    timeout time.Duration
    tls     bool
}

type Option func(*Server)

func WithTimeout(d time.Duration) Option { return func(s *Server) { s.timeout = d } }
func WithTLS() Option                    { return func(s *Server) { s.tls = true } }

func NewServer(addr string, opts ...Option) *Server {
    s := &Server{addr: addr, timeout: 5 * time.Second} // 合理默认值
    for _, opt := range opts {
        opt(s)
    }
    return s
}

// 调用方按需启用，签名稳定，扩展不破坏旧调用
srv := NewServer(":8080", WithTimeout(10*time.Second), WithTLS())
```

> 经验：必填项放普通参数，可选项放 `Option`；不要把所有字段都做成 `Option`，否则反而失去类型约束。

## 四、作用域控制

- **最小作用域**：变量应在最近使用的地方声明，并限制其作用域。
- **优先使用短变量声明 `:=`**：让变量尽量贴近第一次使用的位置。
- **避免提前声明**：函数顶部一次性声明大量变量，会扩大其生命周期与可见范围。

```go
// 不推荐：变量过早声明，作用域过大
func handle(items []Item) error {
    var total int
    var err error
    for _, it := range items {
        total += it.Value
    }
    if total > 100 {
        err = doSomething(total)
    }
    return err
}

// 推荐：变量贴近使用点，作用域最小
func handle(items []Item) error {
    total := 0
    for _, it := range items {
        total += it.Value
    }
    if total > 100 {
        if err := doSomething(total); err != nil {
            return err
        }
    }
    return nil
}
```

尽量保持正常代码路径的缩进最小，并将错误处理缩进，优先处理。这通过允许快速视觉扫描正常路径来提高代码的可读性。例如，不要写：

if err != nil {
    // error handling
} else {
    // normal code
}
Instead, write:  反之，请写：

if err != nil {
    // error handling
    return // or continue, etc.
}
// normal code
If the if statement has an initialization statement, such as:
如果 if 语句有一个初始化语句，例如：

if x, err := f(); err != nil {
    // error handling
    return
} else {
    // use x
}
then this may require moving the short variable declaration to its own line:
然后这可能需要将 short 变量声明移到它自己的行：

x, err := f()
if err != nil {
    // error handling
    return
}
// use x

Named Result Parameters  命名结果参数¶

考虑在 godoc 中会是什么样子。命名结果参数，如：

func (n *Node) Parent1() (node *Node) {}
func (n *Node) Parent2() (node *Node, err error) {}

在 godoc 中会重复；最好使用：

func (n *Node) Parent1() *Node {}
func (n *Node) Parent2() (*Node, error) {}

另一方面，如果一个函数返回两个或三个相同类型的参数，或者结果的意思从上下文中不明确，在某些上下文中添加名称可能是有用的。不要仅仅为了避免在函数内部声明一个变量而命名结果参数；这以牺牲不必要的 API 冗长为代价，换取了轻微的实现简洁性。

func (f *Foo) Location() (float64, float64, error)
is less clear than:
不如：

// Location returns f's latitude and longitude.
// Negative values mean south and west, respectively.
func (f *Foo) Location() (lat, long float64, err error)

如果函数只有几行代码，裸返回是没问题的。一旦函数变得中等规模，请显式地返回值。推论：仅仅因为裸返回而命名结果参数是不值得的。文档的清晰度永远比在函数中节省一两行代码更重要。

最后，在某些情况下，您需要命名一个结果参数，以便在延迟闭包中更改它。这总是可以接受的。

Naked Returns  裸返回
一个不带参数的 return 语句返回命名的返回值。这被称为“裸”返回。

func split(sum int) (x, y int) {
    x = sum * 4 / 9
    y = sum - x
    return
}

Pass Values  传递值
不要仅仅为了节省几个字节而将指针作为函数参数传递。如果函数在调用过程中始终只通过 *x 引用其参数 x ，那么该参数就不应该是指针。这种情况的常见例子包括传递指向字符串（ *string ）的指针或传递指向接口值的指针（ *io.Reader ）。在这两种情况下，值本身都是固定大小的，可以直接传递。这条建议不适用于大型结构体，甚至不适用于那些可能增长的小型结构体。
