# Go 包与文件组织规范

> 包是 Go 工程的最小复用单元，文件则是包内组织代码的物理单位。本文整理「一个包应该如何拆分为多个 `.go` 文件」的常用准则，帮助团队在文件粒度上保持一致、可维护的代码组织方式。

## 一、核心原则

- **包的边界由"对调用者提供的能力"划分**：包内文件的拆分则由"包内部各自承担的职责"划分。
- **优先少而大的包，而非多而散的文件**：拆分文件的目的是降低单文件的认知负担，而不是制造层级。
- **以名词命名文件**：文件名应表达"它装的是什么"，而不是"它做什么动作"。

---

## 二、包的入口文件

每个包都应该以一个 `.go` 文件作为入口，文件名与所在文件夹同名。

例如，`package http` 应放在目录 `http/` 下的 `http.go` 文件中：

```text
http/
└── http.go        // package http 的入口文件
```

这样做的好处：

- 阅读者可以通过同名文件快速定位包的"门面"。
- 当包还很小时，所有内容集中在一处，无需提前规划目录结构。

---

## 三、按职责拆分文件

随着包规模扩大，可以把不同的**职责**拆分到不同文件中，每个文件只负责一块清晰的领域。

以 `net/http` 风格为例：

```text
http/
├── http.go        // 包的入口；底层网络相关逻辑
├── messages.go    // Request / Response 类型，以及网络上的封装、解码
├── client.go      // Client 类型；构造请求等客户端业务逻辑
└── server.go      // Server 类型；路由、处理请求等服务端业务逻辑
```

判断"是否应该拆分"的简单标准：

- 一个文件内已经包含多个不相关的类型或主题。
- 阅读时需要在同一个文件里频繁跳跃才能理解一个完整概念。

> 反过来，**如果几个文件总是被同时修改、且高度相关，可以考虑合并回一个文件**——不要为了拆分而拆分。

---

## 四、根据 import 合并或拆分文件

`import` 块在一定程度上反映了文件的"依赖画像"，可以作为是否合并或拆分文件的参考信号：

- 如果多个文件的 `import` 声明几乎一致，说明它们处理的是同一类问题，**可以考虑合并**。
- 如果一个文件内部出现了明显不同的两组 `import`（比如一半是网络相关、一半是序列化相关），说明它承担了两份职责，**可以考虑按 import 的差异拆开**。

这个准则与「按职责拆分文件」是同一件事的两个视角：职责相似的代码，依赖也通常相似。

---

## 五、文件命名建议

> **Tip**：优先使用名词作为文件名。

推荐与不推荐的命名对照：

```text
// 推荐：名词，描述文件"装的是什么"
messages.go
client.go
server.go
config.go

// 不推荐：动词或动宾短语，更像函数名
handle_request.go
do_init.go
process.go
```

补充建议：

- 不要在文件名中重复包名，例如 `http` 包下不要叫 `http_client.go`，叫 `client.go` 即可。
- 测试文件以 `_test.go` 结尾，与被测文件同名（如 `client.go` 对应 `client_test.go`）。

## 六、关于编译性能的说明

> **Note**：Go 编译器以**包**为单位并行编译；在同一个包内，又会以**函数**为单位并行编译（方法在 Go 里只是带接收者的函数）。

因此：

- **包内文件的拆分方式不会影响编译时间**，可以放心按"可读性"和"职责"来组织文件，而不必担心编译性能。
- 真正影响编译耗时的是包之间的依赖关系，而不是单个包内的文件数量。
