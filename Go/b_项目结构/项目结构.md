# Go 项目结构规范

> 「包」是 Go 工程的最小复用单元，「项目」则是把若干包组合在一起的整体——通常对应一个 Git 仓库或一个 Go module。本文整理 Go 项目层面的组织准则：项目应承担怎样的职责、如何选择包的粒度、如何用 `internal/` 限制 API 暴露面，以及 `main` 包该写到多薄。
>
> **本文倾向 Go 社区主流（Dave Cheney / 标准库 / Kubernetes）的项目结构范式。** 如使用 GoFrame 等基于三层架构（controller / service / dao / model）的强约定框架，请以框架文档为准。

## 一、项目应该有明确的目的

把多个包组合成一个项目时，通常会对应一个 Git 仓库。在 Go 1.11 引入 modules 之后，**module 与 project 基本可以视为同义词**。

每个项目都应该有一个清晰的目的：

- 如果项目是一个**库**，它应当只解决一件事，例如 XML 解析、日志记录。避免把多个无关的功能塞进同一个项目，这样可以躲开「common 库」陷阱。
- 如果项目是一个**应用**（例如 Web 服务、Kubernetes Controller），它的内部可以包含一个或多个 `main` 包。例如 `cmd/contour` 既可以作为 Kubernetes 集群中部署的服务端，也可以作为调试用的客户端。

> **Tip**：在我的经验里，`common` 仓库最终会和它最大的使用者紧紧耦合在一起，从而导致：修复一个 bug 必须同时升级 `common` 与使用者；每次升级都会引入大量无关变更，并且很容易导致 API 破坏。

## 二、选择「少而大」的包

在 Code Review 中，从其他语言转 Go 的程序员最常见的问题之一是：**包拆得太细**。

### 2.1 Go 的可见性只有 public 与 private

Go 并没有提供精细的可见性控制：

- 没有 Java 的 `public` / `protected` / `private` / 隐式 `default`。
- 没有 C++ 的 `friend` 类概念。

Go 中只有两种可见性，通过**标识符首字母大小写**来区分：

| 首字母 | 可见性 | 含义 |
| --- | --- | --- |
| 大写 | public（exported） | 可被任何 Go 包引用 |
| 小写 | private（unexported） | 只能在当前包内部使用 |

> **Note**：你可能会听到有人把 *exported* / *not exported* 当作 *public* / *private* 的同义词使用，二者含义相同。

### 2.2 默认不要创建新包

在可见性控制如此简单的语言里，应该如何避免「过度复杂的包层级」？

**核心建议：优先使用更少、更大的包；默认不要创建新包。**

如果包拆得过细，会导致大量类型不得不暴露为 public，从而形成「又宽又浅」的 API 表面，反而难以维护。

> **Tip**：除了 `cmd/` 和 `internal/`，每个包都应该包含真正的源代码——不要保留空目录或只放一个 `doc.go` 的「占位包」。

如果你来自 Java 或 C# 背景，可以参考下面的类比来「校准」对包粒度的直觉：

> **Tip**：来自 Java 的同学可以这样类比：
>
> - 一个 Java package ≈ 一个 `.go` 源文件。
> - 一个 Go package ≈ 一整个 Maven module 或 .NET assembly。

### 2.3 通过 import 拆分文件，而不是拆分包

如果包是按照「对调用方提供什么能力」来划分的，那么包内部的文件应该如何划分？

判断标准：

- **每个包从一个 `.go` 文件开始**，文件名与目录同名。例如 `package http` 放在 `http/http.go`。
- **随着包变大，按职责拆分文件**：例如 `messages.go` 放 `Request` / `Response` 类型，`client.go` 放 `Client`，`server.go` 放 `Server`。
- **不同文件应承担不同职责**，例如：
  - `messages.go`：HTTP 请求 / 响应在网络上的封包与解包。
  - `http.go`：底层网络处理逻辑。
  - `client.go` / `server.go`：HTTP 业务逻辑（请求构造、路由分发等）。
- **以 `import` 块作为辅助信号**：如果几个文件的 `import` 几乎一致，可考虑合并；如果一个文件中出现明显两组互不相关的 `import`，可按差异拆开。

> **Tip**：**优先使用名词作为源文件名**（如 `client.go`、`messages.go`），避免动词或动宾短语命名（如 `handle_request.go`）。

不必担心「文件拆得多了会拖慢编译」：

> **Note**：Go 编译器以**包**为单位并行编译；在同一个包内，又会以**函数**为单位并行编译（方法在 Go 里只是带接收者的函数）。因此**包内文件的拆分方式不会影响编译时间**，可以放心按可读性和职责来组织。

### 2.4 优先使用 internal 测试

`go` 工具支持把 `testing` 包的测试写在两个位置：

| 测试方式 | 包声明 | 编译归属 | 适用场景 |
| --- | --- | --- | --- |
| internal 测试 | `package http2` | 与被测代码同包，能访问私有标识符 | **单元测试**，直接测试函数 / 方法 |
| external 测试 | `package http2_test` | 独立包，只能调用 public API | `Example` 函数 |

建议：

- **单元测试用 internal 测试**：可以直接测试任意函数 / 方法，避免「为了测试而暴露 API」。
- **`Example` 函数放在 external 测试文件中**：这样在 godoc 中查看时，示例会带有正确的包前缀，方便读者直接复制粘贴。

### 2.5 避免没有 `.go` 文件的中间目录

不要为了"分类"而强行制造包层级。

> **Tip**：除我们后面要讲的 `internal/` 外，Go 包之间的目录层级对 `go` 工具**没有任何意义**。例如 `net/http` 并不是 `net` 的子包或下属包，它们只是路径相邻，地位平等。

如果你发现项目里出现了**只有子目录而没有任何 `.go` 文件的中间目录**，那很可能是把目录当作"分类目录"使用了——这种结构对 Go 没有实际意义，应当尽量避免。

### 2.6 用 `internal/` 收敛公共 API

如果项目里包含多个包，可能会有这种情况：某些函数希望在项目内部跨包使用，但**并不希望暴露给外部使用者**。

Go 工具链为此识别一个特殊的目录名（注意：**是目录名，不是包名**）：`internal/`。

规则：

- 把代码放在某个 `internal/` 目录下，或其子目录中。
- 当 `go` 命令检测到 import 路径中含有 `internal` 时，会验证：进行 import 的包必须位于「以该 `internal` 目录的**父目录**为根」的目录树之内，否则编译失败。

示例：

```text
…/a/b/c/internal/d/e/f
```

- 只能被位于 `…/a/b/c` 目录树下的代码 import。
- `…/a/b/g` 或任何其他仓库都**无法** import 该包。

这样就实现了「**项目内可见，项目外不可见**」的精细可见性控制。

## 三、让 `main` 包尽可能薄

`main` 函数和 `main` 包应该做的事情**越少越好**，原因在于 `main.main` 是一个单例：

- 一个程序（包括测试在内）只能有一个 `main` 函数。
- 因此 `main.main` 调用的所有函数都隐含一个前提：它们只会在 `main.main` 或 `main.init` 期间被调用，且**最多被调用一次**。

这导致写在 `main.main` 里的逻辑非常难以测试。所以应当尽量把业务逻辑从 `main` 函数中剥离出来，理想情况下，把它们从整个 `main` 包中剥离出来。

> **Tip**：`func main()` 应只负责：
>
> - 解析命令行参数 / 配置；
> - 建立基础设施连接（数据库、日志、监控等）；
> - 将后续执行交给一个上层对象（例如 `Server`、`App`）处理。
>
> 业务逻辑应放在可被任意包导入和单元测试的普通包中。
