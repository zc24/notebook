## 一、核心原则

- **全小写 + snake_case**：所有标识符全部小写，单词之间用 `_` 连接。MySQL 在 Linux 下默认对表名大小写敏感（`lower_case_table_names=0`），混用大小写极易导致"在 macOS 能跑、在 Linux 报错"。
- **禁用保留字**：避免 `order`、`user`、`group`、`desc`、`type`、`status` 直接做表名 / 字段名，需要时加业务前缀（如 `t_order`、`user_type`）。完整列表见 [MySQL Reserved Words](https://dev.mysql.com/doc/refman/8.0/en/keywords.html)。
- **禁用中文、拼音、特殊字符**：禁止 `用户表`、`yonghu_biao`、`user@info`，保证跨语言、跨工具（迁移、Dump、ETL）可移植性。
- **见名知意 + 控制长度**：MySQL 标识符上限 64 字符、Oracle 30 字符（旧版）/ 128 字符（19c+）、PostgreSQL 63 字符。建议表名 / 字段名 ≤ 30 字符，给索引前缀留出余量。
- **单数形式优先**：表名建议用单数（`user`、`order`），与 ORM、领域模型保持一致。业界存在分歧（Rails / Laravel 倾向复数），但 **项目内必须统一**。
- **不以数字开头**：`2024_log` 应改为 `log_2024`，避免被解析器当成数字字面量。
- **统一一份命名词表**：比前缀本身更重要的，是团队看到名字能立刻知道它是什么。建议在项目仓库 `docs/db-glossary.md` 中维护业务术语英文对照表。

## 二、库 / 表命名

### 2.1 通用规则

- 数据库（schema）名与项目 / 业务线对齐，如 `order_center`、`user_profile`。
- 表名采用 **`[业务前缀_]主体[_修饰]`** 的结构，主体使用单数名词。
- **不要给所有表加 `t_` / `tbl_` 这类无信息量的统一前缀**（这是 1990 年代 PowerDesigner 时代的习惯），现代实践中只在需要避开 SQL 保留字时才加业务前缀（如 `order` → `t_order`）。
- **关联表**（多对多中间表）：`a_b` 或 `a_b_rel`，按字母顺序排列两端表名，如 `user_role`、`order_coupon`。
- **历史 / 归档表**：在原表名后加 `_history` / `_archive` / `_log`，如 `order_history`、`user_login_log`。
- **分表 / 分区表**：在原表名后加分片后缀，如 `order_0001` ~ `order_1024`、`log_202506`。

### 2.2 命名模板

```text
<business_module>_<entity>[_<modifier>]
```

| 字段 | 必填 | 说明 |
|------|------|------|
| `business_module` | 否 | 业务模块前缀，仅在跨模块同名时使用，如 `order_`、`pay_` |
| `entity` | 是 | 业务实体单数名词，如 `user`、`product` |
| `modifier` | 否 | 修饰后缀，标记表的特殊角色：`_log`、`_history`、`_snapshot`、`_config`、`_relation` |

### 2.3 常见场景示例

业务主表：

```text
user
order
product
coupon
```

关联表（多对多）：

```text
user_role
order_coupon
product_category
```

日志 / 历史 / 快照表：

```text
user_login_log
order_status_history
product_price_snapshot
```

配置 / 字典表：

```text
sys_config
sys_dict
sys_dict_item
```

分表 / 归档：

```text
order_0001         -- 按 user_id 哈希分表
order_202506       -- 按月分表
order_archive      -- 冷数据归档
```

## 三、字段命名

### 3.1 通用规则

- **全小写 snake_case**：`user_name` 而非 `userName` / `UserName` / `user-name`。
- **不与表名重复**：`user` 表里就叫 `name`，不要叫 `user_name`，避免在 SQL 里写出 `user.user_name` 这种"结结巴巴"的列名。
- **禁止使用单字母 / 缩写**：除 `id`、`url`、`ip`、`db` 等业界共识缩写外，避免 `n`、`addr`、`desc`、`amt` 之类自造缩写。
- **外键字段统一以 `_id` 结尾**：指向 `user` 表的外键叫 `user_id`，而不是 `uid` / `userId` / `user`。

### 3.2 推荐字段约定

每张业务表都应包含一组**通用基础字段**，保持全项目一致：

| 字段名 | 类型 | 含义 |
|--------|------|------|
| `id` | `bigint unsigned` | 主键，自增或雪花 ID |
| `created_at` | `datetime` / `timestamp` | 创建时间，由数据库默认值生成 |
| `updated_at` | `datetime` / `timestamp` | 更新时间，由数据库 `ON UPDATE` 维护 |
| `deleted_at` | `datetime` NULL | 软删除标记，`NULL` 表示未删除 |
| `created_by` | `bigint` / `varchar` | 创建人，存用户 ID 或登录名 |
| `updated_by` | `bigint` / `varchar` | 最近更新人 |
| `version` | `int` | 乐观锁版本号 |

> 部分团队历史上习惯用 `gmt_create` / `gmt_modified`（阿里风格）、`create_time` / `update_time`，本规范推荐 `created_at` / `updated_at`，与 Rails、Laravel、Spring Data、GORM 等主流 ORM 默认约定对齐，新项目零摩擦接入。

### 3.3 类型与命名暗示

字段名应能"反推"出类型，降低阅读 SQL 的心智负担：

| 命名模式 | 期望类型 | 示例 |
|---------|---------|------|
| `is_xxx` / `has_xxx` | `tinyint(1)` / `boolean` | `is_deleted`、`has_paid` |
| `xxx_id` | 整型 ID | `user_id`、`order_id` |
| `xxx_at` / `xxx_time` | 时间戳 | `created_at`、`expired_at` |
| `xxx_date` | 日期 | `birth_date`、`settle_date` |
| `xxx_count` / `xxx_num` | 整型计数 | `view_count`、`retry_num` |
| `xxx_amount` / `xxx_fee` | 金额（`decimal`） | `total_amount`、`service_fee` |
| `xxx_rate` / `xxx_ratio` | 比率（`decimal`） | `discount_rate` |
| `xxx_url` | URL 字符串 | `avatar_url`、`callback_url` |
| `xxx_type` / `xxx_status` | 枚举（`tinyint` / `varchar`） | `order_status`、`pay_type` |
| `xxx_json` / `xxx_ext` | JSON / 扩展字段 | `meta_json`、`ext_info` |

> 反例：用 `flag`、`status1`、`temp` 这种无语义命名表达布尔 / 状态，等于把语义写到了"祖传注释"里，新人接手时无从下手。

## 四、索引与约束命名

### 4.1 前缀速查表

这套规范来自 **《阿里巴巴 Java 开发手册》**，国内大厂普遍采用：

| 前缀 | 含义 | 用途 |
|------|------|------|
| `pk_` | Primary Key | 主键（很少显式命名，MySQL 默认就叫 `PRIMARY`） |
| `uk_` | Unique Key | 唯一索引 |
| `idx_` | Index | 普通索引（B-Tree） |
| `ft_` | Fulltext | 全文索引 |
| `fk_` | Foreign Key | 外键（互联网公司一般不用） |

> 阿里规约原文：
>
> 【强制】唯一索引名为 `uk_字段名`；普通索引名为 `idx_字段名`。
> 说明：`uk_` 即 unique key，`idx_` 即 index 的简称。

### 4.2 关键原则

不管选用哪套前缀，**项目内统一**比前缀本身更重要：

1. 同一个库 / 项目里所有表保持一致的命名风格，禁止"一表一风"。
2. 索引名能体现出字段，如 `idx_user_status_created_at` 一眼能看出索引列。
3. 唯一索引一定要带 `uk_` 前缀（或其它能跟普通索引区分的标记），开发看到表结构就知道哪些字段不能重复。
4. 复合索引把字段顺序体现在名字里，例如 `idx_a_b_c` 表示 `(a, b, c)`，便于和 EXPLAIN 中实际命中的索引交叉验证。
5. 不要直接拿字段名当索引名（如 `` KEY `status` (`status`) ``），容易跟字段名混淆，写 `FORCE INDEX` 时尤为痛苦。
6. 索引名长度控制在 **≤ 50 字符**，给 MySQL 64 字符上限留出余量；超长时优先简化字段名而不是去掉前缀。

### 4.3 推荐示例

```sql
-- 主键（默认即可，不用显式命名）
PRIMARY KEY (`id`),

-- 唯一索引
UNIQUE KEY `uk_order_no`              (`order_no`),
UNIQUE KEY `uk_user_type_version`     (`user_id`, `order_type`, `version`),

-- 普通索引（单列）
KEY `idx_status`                      (`status`),
KEY `idx_created_at`                  (`created_at`),

-- 普通索引（复合）
KEY `idx_user_status_created_at`      (`user_id`, `status`, `created_at`),

-- 全文索引
FULLTEXT KEY `ft_content`             (`content`)
```

### 4.4 反例

```sql
-- 不推荐：直接拿字段名当索引名
KEY `created_at` (`created_at`),
KEY `status`     (`status`),

-- 不推荐：风格混乱
KEY `idx_user`           (`user_id`),
KEY `sed_index_status`   (`status`),       -- 自创前缀
KEY `idx_orderType`      (`orderType`),    -- 字段名驼峰，跟其它 snake_case 字段不一致

-- 不推荐：索引名看不出字段
KEY `idx_1` (`a`, `b`),
KEY `idx_2` (`c`),

-- 不推荐：唯一索引和普通索引前缀不分
UNIQUE KEY `idx_username` (`username`),     -- 应该用 uk_username
```

## 五、其他对象（视图 / 存储过程 / 序列）

互联网业务里这类对象使用较少，但一旦用到就应当**前缀化**，避免和表混淆：

| 对象 | 前缀 | 示例 |
|------|------|------|
| 视图（View） | `v_` | `v_user_order_summary` |
| 物化视图（Materialized View） | `mv_` | `mv_daily_revenue` |
| 存储过程（Procedure） | `proc_` | `proc_close_expired_orders` |
| 函数（Function） | `func_` / `fn_` | `func_calc_discount` |
| 触发器（Trigger） | `trg_` | `trg_order_insert_audit` |
| 序列（Sequence，PG / Oracle） | `seq_` | `seq_order_id` |
| 临时表 | `tmp_` | `tmp_import_user_20250619` |

> **关于 `sp_` 前缀**：在 **SQL Server** 中 `sp_` 是系统存储过程的保留前缀（系统过程位于 `master` 库），用户过程若以 `sp_` 开头，引擎会先去 `master` 查找再回到当前库，带来额外开销且可能与未来新增的系统过程重名，微软官方明确建议[避免使用](https://learn.microsoft.com/en-us/sql/relational-databases/stored-procedures/create-a-stored-procedure)。**MySQL / PostgreSQL / Oracle 不存在此机制**，`sp_` 在功能上完全无副作用，但出于跨库迁移和团队语义统一考虑，本规范统一推荐使用 `proc_`。

## 六、大厂 / 开源项目对照

| 项目 / 公司 | 表命名 | 字段命名 | 索引命名 | 备注 |
|------------|--------|---------|---------|------|
| **阿里巴巴 Java 开发手册** | 单数 + 业务前缀 | snake_case | `pk_` / `uk_` / `idx_` | 国内最广为遵循的一套 |
| **GitLab** | 单数 snake_case | snake_case | `index_<table>_on_<columns>` | Rails 默认风格 |
| **Ruby on Rails / Laravel** | 复数 snake_case | snake_case | `index_<table>_on_<columns>` | ORM 自动生成 |
| **Django** | 单数 snake_case（默认 `<app>_<model>`） | snake_case | `<app>_<model>_<col>_<hash>` | 自动加 8 位 hash 后缀 |
| **GORM / Spring Data JPA** | 单数 snake_case | snake_case | `idx_<col>` / `uk_<col>` | 与阿里规约最接近 |
| **PostgreSQL 官方建议** | 单数 snake_case | snake_case | `<table>_<col>_idx` / `<table>_<col>_key` | 后缀风格 |
| **Shopify / Stripe（业界博客）** | 单数 snake_case | snake_case | `index_<table>_on_<col>` | 类 Rails 风格 |

> **结论**：业界主流分两派——**前缀派**（`idx_xxx`，国内 + Java 系）和 **后缀派**（`xxx_idx`，PG + Rails 系）。本规范推荐前缀派，与国内绝大多数 MySQL 项目和阿里规约保持一致。

## 七、常见反例

表 / 字段层面：

| 反例 | 问题 | 正确写法 |
|------|------|---------|
| `UserInfo` | 大驼峰 + 跨平台风险 | `user_info` |
| `t_userInfo` | 前缀风格 + 字段风格混用 | `user_info` |
| `用户表` / `yonghu_biao` | 中文 / 拼音 | `user` |
| `order` | 撞 SQL 保留字 | `t_order` 或 `customer_order` |
| `tb_user`、`tbl_user`、`t_user` 全项目随机出现 | 前缀不统一 | 团队统一选一种 |
| `userName` / `createTime` | 驼峰命名 | `user_name` / `created_at` |
| `flag` / `status1` / `temp` | 无语义字段名 | `is_deleted` / `order_status` |
| `2024_order_log` | 数字开头 | `order_log_2024` |
| `uid` 指向 `user.id`、`u_id` 指向另一张表 | 外键命名不一致 | 统一用 `user_id` |

索引层面：

```sql
-- 反例：索引名 = 字段名，FORCE INDEX 时极易混淆
KEY `status` (`status`),
-- 正例：
KEY `idx_status` (`status`),

-- 反例：索引名无语义，无法从名字推出索引列
KEY `idx_1` (`a`, `b`),
-- 正例：
KEY `idx_a_b` (`a`, `b`),

-- 反例：唯一索引用了普通索引前缀
UNIQUE KEY `idx_email` (`email`),
-- 正例：
UNIQUE KEY `uk_email` (`email`),

-- 反例：字段名是驼峰，跟其它 snake_case 字段不一致
KEY `idx_orderType` (`orderType`),
-- 正例：
KEY `idx_order_type` (`order_type`),
```

## 八、工程化落地建议

1. **在 `docs/db-style-guide.md` 中明确规范**：把本规范精简版写入项目仓库，作为新成员入职必读、Code Review 必查项。
2. **CI 中强制校验 DDL**：用 [sqlfluff](https://docs.sqlfluff.com/)、[squawk](https://squawkhq.com/)、[Atlas](https://atlasgo.io/) 等工具在 PR 时校验迁移脚本：
   - 表名 / 字段名是 snake_case；
   - 索引名带 `idx_` / `uk_` 前缀；
   - 不允许使用 SQL 保留字作表名；
   - 不允许新增不带索引的外键关联字段（FK 列必须有索引）。
3. **DB Schema 由迁移工具管理**：所有 DDL 通过 [Flyway](https://flywaydb.org/) / [Liquibase](https://www.liquibase.org/) / [golang-migrate](https://github.com/golang-migrate/migrate) / [gh-ost](https://github.com/github/gh-ost) 等工具版本化，禁止线上手工改表。
4. **建表模板化**：在项目里维护一份 `templates/create_table.sql`，预置 `id` / `created_at` / `updated_at` / `deleted_at` 等基础字段、字符集、注释、索引前缀，新人复制改改就能用。
5. **强制写 `COMMENT`**：表和字段都要写中文注释，DBA 工具（Navicat / DataGrip / DBeaver）和文档生成器（如 [SchemaSpy](https://schemaspy.org/)）会自动展示，省去维护单独"字典表"。
6. **配合 ORM 自动生成实体**：用 GORM `gen` / MyBatis `generator` / Prisma 等工具从 DDL 反向生成实体类，避免手写字段时和表结构对不上。
7. **保护生产库 Schema**：生产环境禁用 `DROP TABLE` / `ALTER TABLE` 直连权限，所有变更走"工单 → DBA 评审 → gh-ost 在线变更 → 双跑校验"链路。
8. **定期 Schema 走查**：每季度跑一次脚本检查命名规范、缺失索引、未使用索引、字段超宽（如 `varchar(4000)`），形成"DB 健康度报告"。
