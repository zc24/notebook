# 数据库索引命名规范

## 一、最主流的命名约定（国内大厂普遍采用）


| 前缀     | 含义          | 用途                              |
| ------ | ----------- | ------------------------------- |
| `pk_`  | Primary Key | 主键（很少显式命名，MySQL 默认就叫 `PRIMARY`） |
| `uk_`  | Unique Key  | 唯一索引                            |
| `idx_` | Index       | 普通索引（B-Tree）                    |
| `ft_`  | Fulltext    | 全文索引                            |
| `fk_`  | Foreign Key | 外键（互联网公司一般不用）                   |


这套规范来自 **《阿里巴巴 Java 开发手册》**（俗称"阿里规约"），原文：

> 【强制】唯一索引名为 `uk_字段名`；普通索引名为 `idx_字段名`。
> 说明：uk_ 即 unique key，idx_ 即 index 的简称。

## 二、关键原则（比前缀本身更重要）

不管用哪套，关键是 **项目内统一**：

1. 同一个库 / 项目里所有表保持一致的命名风格
2. 索引名能体现出字段（如 `idx_ptype_status_dbctime`）
3. 唯一索引一定要带 `uk`_ 前缀（或其它能跟普通索引区分的标记），开发看到表结构就知道哪些字段不能重复
4. 复合索引把字段顺序体现在名字里，例如 `idx_a_b_c` 表示 `(a, b, c)`
5. 不要直接拿字段名当索引名（如 `KEY \`status (status)`），容易跟字段名混淆

## 三、推荐命名示例

```sql
-- 主键（默认即可，不用显式命名）
PRIMARY KEY (`id`),

-- 唯一索引
UNIQUE KEY `uk_username`              (`username`),
UNIQUE KEY `uk_ptype_version_time`    (`ptype`, `version`, `start_time`),

-- 普通索引（单列）
KEY `idx_status`                      (`status`),
KEY `idx_dbctime`                     (`dbctime`),

-- 普通索引（复合）
KEY `idx_ptype_status_dbctime`        (`ptype`, `status`, `dbctime`),

-- 全文索引
FULLTEXT KEY `ft_content`             (`content`)
```

## 四、反面示例

```sql
-- 不推荐：直接拿字段名当索引名
KEY `dbctime` (`dbctime`),
KEY `status`  (`status`),

-- 不推荐：风格混乱
KEY `idx_ptype`          (`ptype`),
KEY `sed_index_status`   (`status`),       -- 自创前缀
KEY `idx_spiderSource`   (`spiderSource`), -- 字段名驼峰，跟其它 snake_case 字段不一致

-- 不推荐：索引名看不出字段
KEY `idx_1` (`a`, `b`),
KEY `idx_2` (`c`),
```
